TL;DR

🧠 One-line mental model: JavaScript = a system of boxes holding values + one thread that does one thing at a time.

You only ever do three things: ① store values (variables, objects, arrays), ② write rules that process them (operators, conditionals, loops, functions), and ③ react to the outside world (DOM events, network calls). The third is where async comes from — and where 90% of beginner confusion lives.

The recommended order (also this article's chapters): values & types → control flow → functions & closures → objects & arrays → DOM & events → async → modules & traps. Don't skip. If chapter 3 (functions) doesn't click, chapter 6 (async) will be a wall.

★ The fastest way to learn JS is to type while you read. Hit F12 to open the browser Console — every code block here can be pasted in and run. One run beats ten reads.

0 · Before You Start — What JavaScript Is and Where It Runs

0.1 What JavaScript actually is

JavaScript is an interpreted programming language. It began life inside browsers to make web pages move. Today it runs in three places:

EnvironmentWhat you buildExtra powers
BrowserWeb interactivity, the whole frontend ecosystem (React, Vue…)DOM, fetch, localStorage
Node.jsBackend APIs, CLI tools, build scriptsFile system, HTTP servers
ElsewhereDesktop apps (Electron), mobile (React Native)Platform-specific

★ The language core is identical everywhere (chapters 1–4, 6–7). Only chapter 5, the DOM, is browser-only. Learn the core well and switching environments costs you nothing.

0.2 Write your first line in ten seconds — nothing to install

On this very page press F12 (Mac: Cmd + Option + I) → open the Console tab → type → Enter. That's it, you're writing JavaScript.

// Paste into the Console and press Enter
console.log("Hello, JavaScript!");

// It's also a calculator
2 + 3 * 4      // → 14

To make it part of a page, load the file with a <script> tag. Always add defer — the browser then parses all the HTML before running your JS, so you never hit "the element doesn't exist yet".

<!DOCTYPE html>
<html>
<head>
  <!-- defer: runs after HTML is parsed, and never blocks rendering -->
  <script src="main.js" defer></script>
</head>
<body>
  <h1 id="title">Hello</h1>
</body>
</html>

1 · Values & Types — What Data Looks Like

1.1 Variables: use const and let, forget var

A variable is just a name for a value so you can refer to it later.

const name = "Michael";   // constant: never reassigned → use this by default
let count = 0;             // variable: you plan to reassign it
count = count + 1;         // ✅ let allows reassignment

name = "Someone";          // ❌ TypeError: Assignment to constant variable
var old = 1;               // ❌ legacy; its scoping rules bite. Don't.

The rule is simple: always write const first, and only downgrade to let when you know you'll reassign. This isn't fussiness — const tells every reader (including you in three months) "this name never changes".

constletvar (avoid)
ReassignableNoYesYes
ScopeBlock { }Block { }Whole function (error-prone)
Used before declaredThrows (good)Throws (good)Gives undefined (bad)

★ const obj = {…} means "the name obj won't point somewhere else", not "the contents are frozen". obj.age = 30 is perfectly legal. The next section's diagram makes this click.

1.2 Eight types, but you use five daily

// ── Primitives: the value itself ──
const n = 42;              // number  (no int/float split — one number type)
const s = "hi";            // string  (single or double quotes; backticks for templates)
const b = true;            // boolean (true / false)
let   u;                   // undefined (declared, never assigned)
const e = null;            // null    (deliberately empty)

// ── Objects: a bundle of things ──
const obj  = { a: 1 };     // object
const arr  = [1, 2, 3];    // arrays are objects too
const fn   = () => {};     // so are functions

// Check a type with typeof
typeof 42        // "number"
typeof "hi"      // "string"
typeof null      // "object" ← a historical language bug; just memorise it
Array.isArray(arr) // true  ← the correct way to detect an array

The string trick you'll use constantly — template literals, written with backticks `, interpolate variables and span lines:

const user = "Michael", age = 30;

// ❌ old style: a pile of plus signs, hard to read
"Hi " + user + ", you are " + age + " years old."

// ✅ template literal
`Hi ${user}, you are ${age} years old.`

1.3 The single most important diagram: primitives vs objects

Here's the first real trap: copying a number and copying an object behave nothing alike.

// Primitive: copied by value
let a = 1, b = a;  b = 2;
console.log(a);            // 1  ← untouched

// Object: copied by reference
let x = { n: 1 }, y = x;  y.n = 2;
console.log(x.n);          // 2  ← it changed too!

// ✅ Want a genuine copy? Spread makes a shallow one
let z = { ...x };  z.n = 99;
console.log(x.n);          // 2  ← this time unaffected

// Nested objects need a deep copy (built into modern browsers)
const deep = structuredClone(x);

★ This one rule ripples all the way into React state updates, array sorting and argument passing. Spend five minutes on it now, save ten hours of debugging later.

1.4 Operators and truthiness — === always beats ==

JavaScript silently converts types for you (type coercion), which is its most notorious feature. The fix is trivial: always compare with ===.

"1" == 1       // true  ← == converts first, then compares. Unpredictable.
0 == false     // true
null == undefined // true

"1" === 1      // false ← different types → false. Predictable.
0 === false    // false

NaN === NaN     // false ← the one exception; test with Number.isNaN(x)

There are exactly six falsy values — everything else is truthy. Memorise them, because you write if (x) every day:

// These six are false inside an if:
false, 0, "", null, undefined, NaN

// ⚠ These "look empty" but are TRUE!
"0"  []  {}  // empty string is false, but empty array/object are true

Three modern operators that delete a lot of if-statements:

const user = { profile: { name: "Michael" }, score: 0 };

// ?. optional chaining: returns undefined instead of exploding
user.address?.city        // undefined (no error)
user.address.city          // ❌ TypeError

// ?? nullish coalescing: falls back ONLY on null/undefined
user.score ?? 100         // 0   ← 0 is a real value, kept
user.score || 100         // 100 ← ❌ 0 was falsy and got clobbered. Classic bug.

// Ternary: a one-line if/else
const label = user.score > 60 ? "pass" : "fail";

2 · Control Flow — Branching and Repeating

2.1 Conditionals

const score = 75;

if (score >= 90) {
  console.log("A");
} else if (score >= 60) {
  console.log("Pass");
} else {
  console.log("Fail");
}

// When there are many branches all testing equality, switch reads better
switch (action) {
  case "add":    doAdd();    break;  // ⚠ forget break and it falls through
  case "remove": doRemove(); break;
  default:        doNothing();
}

★ Nested ifs more than two deep should be rewritten. The technique is early return: reject the invalid cases up front with return, and the main logic stays at the outermost level where it's easy to read.

2.2 Loops: three is enough

const fruits = ["apple", "banana", "cherry"];

// ① for...of — walk an array. Correct 90% of the time.
for (const f of fruits) console.log(f);

// ② for — when you need the index, or custom stepping / reverse order
for (let i = 0; i < fruits.length; i++) console.log(i, fruits[i]);

// ③ while — unknown count; a condition decides when to stop
let n = 1;
while (n < 100) n = n * 2;

// Iterate an object's key/value pairs
for (const [key, value] of Object.entries({a:1, b:2})) console.log(key, value);

// Two control keywords:
break;     // stop the whole loop
continue;  // skip this round, carry on with the next

★ In practice you'll reach for map / filter / reduce (chapter 4) far more often than a hand-written loop. But loops are the foundation — know them.

3 · Functions & Closures — The Heart of the Language

A function wraps up a piece of logic, gives it a name, and lets you feed different values through it. If this chapter doesn't land, chapter 6 will hurt.

3.1 Three ways to write one

// ① Function declaration — hoisted, callable before its definition
function add(a, b) { return a + b; }

// ② Function expression — stored in a variable, must be defined first
const sub = function (a, b) { return a - b; };

// ③ Arrow function — the modern default: short, with simple `this` behaviour
const mul = (a, b) => a * b;        // single expression → implicit return
const greet = name => `Hi ${name}`;  // one param → parens optional
const log = (msg) => {              // with braces you must return yourself
  console.log(msg);
  return msg.length;
};

Two parameter tricks worth knowing:

// Default parameters
function greet(name, greeting = "Hello") { return `${greeting}, ${name}!`; }
greet("Michael");            // "Hello, Michael!"

// Rest parameters: collect the remainder into an array
function sum(...nums) { return nums.reduce((a, b) => a + b, 0); }
sum(1, 2, 3, 4);            // 10

3.2 Scope and closures — the classic interview question, and genuinely intuitive

Scope is the region where a variable is visible. Inner code can see outer variables; outer code cannot see inner ones.

const APP = "site";             // global

function makeCounter() {
  let count = 0;               // local to makeCounter
  return function increment() {
    count++;                   // ← remembers the outer count. That's a closure.
    return count;
  };
}

const c1 = makeCounter();
c1(); c1();                    // 1, 2
const c2 = makeCounter();
c2();                          // 1  ← a fresh one, unrelated to c1

console.log(count);            // ❌ ReferenceError: invisible from outside
Interactive: two counters, each remembering its own count
0
counter 1
0
counter 2
These buttons call two independent closures produced by the same makeCounter — each keeps its own count, and they can never contaminate each other.

Where closures earn their keep: private variables nobody outside can touch, event handlers that remember the data they were created with, and — in chapter 6 — async callbacks that still reach their original context.

3.3 Functions are values — higher-order functions

In JS a function is itself a value, so it can be passed as an argument and returned as a result. That single rule underpins all modern JavaScript style.

// Pass a function in (a callback)
[1, 2, 3].map(n => n * 2);          // [2, 4, 6]
button.addEventListener("click", () => alert("hi"));

// Return a function (a factory)
const multiplier = factor => n => n * factor;
const double = multiplier(2);
double(5);                        // 10

4 · Objects & Arrays — The Shape of Real Data

4.1 Objects: related values under names

const user = {
  name: "Michael",
  age: 30,
  tags: ["js", "react"],
  greet() { return `Hi, ${this.name}`; }   // a function on an object is a "method"
};

user.name;            // "Michael"   dot notation (preferred)
user["name"];        // "Michael"   brackets only when the key is a variable
user.email;           // undefined   reading a missing key doesn't throw
user.email = "a@b.c"; // add on the fly
delete user.age;      // remove
"name" in user;      // true  does this key exist?

Object.keys(user);    // ["name","tags","greet","email"]
Object.entries(user); // [["name","Michael"], …] handy for looping

Destructuring and spread — modern JS uses these on nearly every line, so get fluent:

// Destructuring: pull out the fields you want in one go
const { name, age = 18 } = user;        // default when age is missing
const [first, second] = ["a", "b"];      // works on arrays too

// Destructured parameters: callers stop caring about argument order
function draw({ x = 0, y = 0, color = "red" }) { /* … */ }
draw({ color: "blue", x: 10 });

// Spread: copy + merge (later keys win)
const updated = { ...user, age: 31 };   // new object; the original user is untouched
const merged  = [...arr1, ...arr2];

4.2 Array methods — learn these seven, write half the loops

The dividing line that matters: does it mutate the original array? Mutating methods are a rich source of bugs.

MethodWhat it doesReturnsMutates?
mapTransform every elementNew array (same length)No
filterKeep the ones that matchNew array (shorter)No
findFirst matchThe element / undefinedNo
reduceFold the array into one valueAnythingNo
▶ reduce((acc, cur) => …, initial): acc the accumulator, the previous round's return value (always return it, or the next round gets undefined) · cur the current element · initial the seed value, always pass it, an empty array with no seed throws
some / everyAny match / all matchbooleanNo
forEachRun a side effect per elementundefinedNo
push/pop/sort/reverse/spliceAdd, remove, reorderVariesYes ⚠
const nums = [1, 2, 3, 4, 5];

nums.map(n => n * 2);              // [2,4,6,8,10]
nums.filter(n => n % 2 === 0);     // [2,4]
nums.find(n => n > 3);             // 4
nums.reduce((sum, n) => sum + n, 0); // 15  (0 is the seed — always pass it)
nums.some(n => n > 4);             // true
nums.every(n => n > 0);            // true
nums.forEach(n => console.log(n));  // prints 1,2,3,4,5; returns undefined

// Chain them like an assembly line — the nicest part of JS
const users = [
  { name: "A", age: 17, active: true  },
  { name: "B", age: 25, active: true  },
  { name: "C", age: 30, active: false }
];
const names = users
  .filter(u => u.active && u.age >= 18)   // narrow first
  .map(u => u.name)                        // then transform
  .join(", ");                          // "B"

// ⚠ sort mutates AND sorts as strings by default!
[10, 9, 1].sort()                  // [1, 10, 9] ← not what you wanted
[10, 9, 1].toSorted((a,b) => a-b)  // [1, 9, 10] ← comparator, and no mutation

★ The roles in toSorted's comparator: toSorted((a, b) => …). a, b are two elements being compared; the number you return decides the order: negative → a goes first, positive → a goes after, 0 → leave them. So (a,b) => a-b is exactly numeric order. (With no comparator, sort() converts elements to strings first, so "10" comes before "9" — char '1' < '9'.) Use toSorted over sort to not mutate the original (it returns a sorted copy).

5 · DOM & Events — Making the Page Move

Everything so far was the language core, identical in Node.js. This chapter is browser-only: the DOM is the tree the browser builds from your HTML, and JS reads and writes the page through it.

5.1 Selecting and changing elements

// Select: two methods are all you need — they take CSS selectors
const el   = document.querySelector("#title");        // first match
const all  = document.querySelectorAll(".item");       // all matches (NodeList)

// Change content
el.textContent = "New title";    // ✅ plain text, safe, the default choice
el.innerHTML = userInput;       // ⚠ parsed as HTML — user input here is an XSS hole

// Change styling: toggle classes rather than writing inline styles
el.classList.add("active");
el.classList.toggle("open");
el.style.color = "red";         // use sparingly — styling belongs in CSS

// Attributes and data
el.setAttribute("aria-label", "Close");
el.dataset.id;                    // maps to data-id="…" in the HTML

// Create elements
const li = document.createElement("li");
li.textContent = "New item";
document.querySelector("#list").appendChild(li);
li.remove();                      // delete itself

    5.2 Events: catching what the user does

    button.addEventListener("click", (e) => {
  console.log("clicked", e.target);
});

// Form submit: always cancel the default full-page reload
form.addEventListener("submit", (e) => {
  e.preventDefault();
  const data = Object.fromEntries(new FormData(form));
  console.log(data);
});

// The events you'll actually use: click / input / change / submit / keydown / scroll

    Event delegation — every frontend dev uses this. Don't attach 100 listeners to 100 buttons; attach one to their shared parent. It even covers elements added later.

    // ❌ one listener each: slow at scale, and new elements get nothing
document.querySelectorAll(".item").forEach(el => el.addEventListener("click", …));

// ✅ one listener on the parent, catching everything via bubbling
list.addEventListener("click", (e) => {
  const item = e.target.closest(".item");   // walk up to the nearest .item
  if (!item) return;                        // clicked empty space — ignore
  console.log("clicked", item.dataset.id);
});

    6 · Async — The Biggest Hurdle

    JavaScript is single-threaded: one thing at a time. So slow work like "wait for the network" can't block the thread — it's handed to the browser, and you get called back when it's done. That's the entire motivation for async.

    6.1 Understand the event loop and the rest follows

    console.log("1");
setTimeout(() => console.log("2"), 0);   // even with a 0ms delay
Promise.resolve().then(() => console.log("P"));
console.log("3");

// Output order: 1 → 3 → P → 2
// All sync code first → microtasks (Promises) → macrotasks (setTimeout)

    6.2 Promises: a receipt for a future value

    A Promise is an object representing a result that isn't here yet. Three states: pending → fulfilled or rejected. Once settled, it never changes again.

    // You'll rarely write `new Promise` — usually you consume one (e.g. from fetch)
fetch("/api/user")
  .then(res => res.json())          // success → next link in the chain
  .then(data => console.log(data))
  .catch(err => console.error(err)) // any failure in the chain lands here
  .finally(() => setLoading(false)); // runs either way

    6.3 async / await: async code that reads like sync code (use this)

    // Same job, but it reads top to bottom
async function loadUser(id) {
  try {
    const res  = await fetch(`/api/user/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`); // ⚠ fetch only rejects on network failure — a 404 does NOT!
    const user = await res.json();
    return user;
  } catch (err) {
    console.error("load failed:", err);
    return null;
  }
}

// Calling it: an async function always returns a Promise, so await (or .then) it
const user = await loadUser(1);

    The three mistakes every beginner makes:

    // ❌ 1: forgetting await — you get a Promise object, not the data
const user = loadUser(1);
console.log(user.name);   // undefined — user is a Promise{}

// ❌ 2: queuing work that could run together (3 seconds instead of 1)
const a = await fetchA();   // wait 1s
const b = await fetchB();   // wait another 1s
const c = await fetchC();   // and another → 3s total

// ✅ independent calls: fire together, Promise.all waits for all of them
const [a, b, c] = await Promise.all([fetchA(), fetchB(), fetchC()]); // 1s total

// ❌ 3: await inside forEach — it waits for nothing at all
items.forEach(async (i) => { await save(i); });  // the loop doesn't wait
for (const i of items) { await save(i); }        // ✅ sequential
await Promise.all(items.map(i => save(i)));      // ✅ parallel
    Interactive: sequential await vs Promise.all (three 1-second requests)

    Three independent requests. Fired together with Promise.all, the wall clock drops from 3 seconds to 1.

    7 · Modules and the Traps Everyone Hits

    7.1 Modules: splitting code across files

    // ── math.js ──
export function add(a, b) { return a + b; }   // named export (many per file)
export default class Calculator { }              // default export (one per file)

// ── main.js ──
import Calculator, { add } from "./math.js";    // the extension is required

// ── index.html ── type="module" is mandatory
<script src="main.js" type="module"></script>

    ★ A type="module" file is automatically strict mode, automatically deferred, and scoped to itself (no global pollution). Modules also can't be opened over file:// — you need a local server; the quickest is npx serve.

    7.2 Six traps you will hit

    1. this differs in arrow functions
    A normal function's this depends on who called it; an arrow function has no this and borrows the enclosing one. Normal functions for object methods, arrows for callbacks — do that and you'll almost never be wrong.
    obj.method(){…} ✓　callback: () => {} ✓
    2. Floats are imprecise
    0.1 + 0.2 !== 0.3 (it's 0.30000000000000004). Not a JS bug — every language's floats do this.
    Store money as integer cents, never decimals
    3. Reading async results too early
    Outside the .then / await, the value is undefined. Use the data where you receive it.
    A missing await is the single most common bug
    4. Array methods that mutate
    sort／reverse／splice change in place. Copy first, or use the newer toSorted／toReversed.
    [...arr].sort() or arr.toSorted()
    5. innerHTML with user input
    That's an open door for XSS. For plain text, always textContent.
    textContent is the safe default
    6. Comparing with ==
    Nobody remembers the coercion rules. Use ===. The one useful exception: x == null catches null and undefined together.
    Turn on ESLint's eqeqeq rule and forget about it

    Trap 1, expanded　how this gets lost, and the fix

    const counter = {
  count: 0,
  start() {                       // method = normal function → this = counter
    setInterval(() => {           // callback = arrow → keeps the outer this
      this.count++;
      console.log(this.count);    // 1, 2, 3…
    }, 1000);
  }
};

// ❌ common mistake: callback as a normal function, and this is gone
setInterval(function () { this.count++; }, 1000);  // this isn't counter → NaN

    Trap 2, expanded　how floats are usually handled

    0.1 + 0.2;                 // 0.30000000000000004

// Display: round to the digits you need
(0.1 + 0.2).toFixed(2);              // "0.30" (string)
Math.round((0.1 + 0.2) * 100) / 100;    // 0.3 (number)

// Compare: use a tolerance, never ===
Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON;   // true

// Money: store and compute in integer cents, divide back only at the end
const cents = 1010 + 2050;                // 3060 cents
`$${(cents / 100).toFixed(2)}`;             // "$30.60"
