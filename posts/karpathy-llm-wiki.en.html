LLM Wiki · Let an LLM Maintain a Knowledge Base That Grows

Instead of re-retrieving documents on every question (RAG), the LLM acts like a wiki editor: it writes what it reads into a persistent, interlinked set of markdown files — and next time it just reads the wiki.
The real point: it solves not a retrieval-technology problem, but the willpower problem of maintaining a knowledge base over time.
Source: Andrej Karpathy · GitHub Gist ↗ Type: design pattern / idea file Compiled: 2026-07-16

TL;DR

📚 LLM Wiki shifts retrieval cost from query time to write time — reason once, bake it into a page, then reads replace re-computation.

Classic RAG re-retrieves and re-assembles "the shared conclusion of these five docs" every single time. LLM Wiki has the model write that synthesis into one markdown page, and proactively update cross-references and flag contradictions. Humans curate sources, set direction, judge what matters; the LLM does all the bookkeeping grunt work.

★ Karpathy explicitly frames this as an idea file you can hand to your agent to co-develop — not a spec. His closing line: "Everything mentioned above is optional and modular." The value is a pattern, not a product.

The problem it solves

Karpathy's core claim is a single sentence:

The tedious part of maintaining a knowledge base is not the reading or the thinking — it's the bookkeeping.

People abandon their wikis not for lack of insight, but because updating cross-references, keeping summaries current, and flagging stale claims scales with size until it crushes the will to maintain it. An LLM doesn't get bored, doesn't forget to update links, and can edit 15 files at once — so maintenance cost approaches zero.

He compares it to Vannevar Bush's 1945 Memex: Bush wanted a private, actively-organized store where the links are as valuable as the documents. The one thing he never solved was who maintains it — the LLM fills that gap.

Three-layer architecture

① Raw Sources (immutable)
Original documents — articles, papers, images, data. The LLM reads but never edits them; the single source of truth.
② The Wiki (LLM-generated)
Interlinked markdown: summaries, entity pages, comparisons, synthesis pages. The LLM fully owns this layer — creating, updating, maintaining links. Humans read, the LLM writes.
③ The Schema (config)
CLAUDE.md / AGENTS.md: defines wiki structure, conventions, workflows. Turns a general chatbot into a disciplined wiki editor.

The loop: Schema sets the rules → the LLM distills Raw Sources into the Wiki → humans explore the Wiki and, when rules fall short, go back and edit the Schema. This upgrades "prompt engineering" into a living, persistent agent-governance document.

Three core operations

Ingest

Drop in a new source → LLM reads → discusses key points with you → writes a summary page → updates the index → goes back and edits related entity/concept pages (one source often touches 10–15 pages) → appends to log.md. The most expensive and error-prone operation.

Query

Read the index to find relevant pages → read them → synthesize an answer with citations, output as a markdown page / comparison table / slides / chart. Key insight: good answers get "filed back into the wiki" as new pages — ad-hoc analysis shouldn't live only in chat history. This makes querying and writing the same action.

Lint

Periodically have the LLM check for: contradictions across pages, stale claims that a newer source overturned, orphan pages with no inbound links, important concepts mentioned but lacking their own page, missing cross-references, gaps fillable by web search. The step most easily skipped — and the one the community agrees you can least afford to skip.

Implementation details (no database needed)

ComponentWhat it does
index.mdA content-oriented catalog: one-line summary per page + categories. Queries read it first. Karpathy notes it's enough up to ~100 sources / a few hundred pages — no embedding-based RAG infrastructure required.
log.mdAppend-only timeline with a fixed prefix ## [2026-04-02] ingest | title, parseable by plain unix tools (grep "^## \[" log.md | tail -5).
GitThe wiki is a git repo — history, branching, collaboration for free. The core reason to pick markdown over a database.
ObsidianUsed as the "IDE": graph view for topology (who's a hub, who's an orphan), Web Clipper to turn pages into markdown, Dataview queries over frontmatter, Marp to generate slides directly.
qmd (optional)Bolt on as it scales: local BM25 + vector hybrid search + LLM re-rank, CLI or MCP. By Shopify CEO Tobi Lütke, ~28k GitHub stars — a real, popular tool.

An engineer's take

👍 What's good

  • Trading write amplification for query quality: cross-document reasoning is done once and baked into a page; queries become reads. A classic database trade-off — just with an LLM as the ETL.
  • Zero-infrastructure start: markdown + git + grep is all you need — no vector DB, no embedding pipeline. For solo PKM / small research, friction is minimal.
  • A 30-year-old metaphor for human/machine division: humans judge, the LLM does bookkeeping. "Wiki editor" is intuitive enough that non-engineers grasp "I'm working with an assistant that never gets tired."

KP1 · It's write-time ETL
Cross-doc reasoning shifted to write time; queries become reads. Write amplification bought with query quality.
KP2 · Query = writing
Good answers get filed back as new pages — the biggest behavioral difference from RAG.
KP3 · Don't skip lint
Karpathy lists it as optional; the community says it's the one step you can't skip — no lint, the wiki drifts and rots.
KP4 · It solves willpower, not tech
It addresses humans abandoning knowledge-base upkeep, not a retrieval-algorithm problem.

When to use it, when not to

Good fit: personal research PKM, long-running topic tracking (deep industry/company research), a reading companion. These have "queries far more frequent than writes, and tolerate a little drift" — the sweet spot. (This very site runs a stock wiki that's an instance of the same pattern.)

Poor fit: anything needing strict traceability or regulatory compliance (finance/medical decision basis) that can't tolerate an LLM summary layer between source and conclusion — unless paired with version-pinned citations; and one-off tasks you'll never re-query, which don't justify ingest's write cost.
