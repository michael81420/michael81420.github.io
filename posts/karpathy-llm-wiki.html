LLM Wiki · 讓 LLM 幫你維護一份會長大的知識庫

不是每次問問題都重新翻資料（RAG），而是讓 LLM 像維基管理員一樣，把每次讀到的東西寫進一份持久、互相連結的 markdown 知識庫——下次直接讀維基。
口號本質：解決的不是檢索技術，而是人類長期維護知識庫的意志力問題。
來源:Andrej Karpathy · GitHub Gist ↗ 性質:設計模式 / idea file 整理日:2026-07-16

TL;DR

📚 LLM Wiki 把「檢索的成本」從查詢時攤到寫入時——推理做一次、寫死進頁面，之後查詢是讀取而非重算。

傳統 RAG 每次問「這五份文件的共同結論」都要重新檢索、重新拼裝。LLM Wiki 讓 LLM 把綜合結果寫成一頁 markdown，並主動更新交叉引用、標記矛盾。人負責找資料、下方向、判斷意義；LLM 負責所有「記帳型苦力」。

★ Karpathy 明確定位這是可貼給 agent 一起發展的 idea file，不是規格書。文末原話：「以上一切都是選用且模組化的。」文章價值在提出「模式」，不是提出「產品」。

它解決什麼問題

Karpathy 的核心論點只有一句：

維護知識庫最累的不是閱讀或思考，而是記帳（bookkeeping）。

人類維護 wiki 會半途而廢，不是缺乏洞見，而是「更新交叉引用、保持摘要最新、標記過時資料」這種苦力活隨規模成長，最終壓垮維護意願。LLM 不會嫌煩、不會忘記更新連結、可以一次改 15 個檔案——維護成本趨近於零。

他把這比作 Vannevar Bush 1945 年的 Memex 構想：Bush 想要一個「私人、被主動整理、連結和文件本身一樣有價值」的知識庫，唯一沒解決的是「誰來維護」——LLM 補上了這塊。

傳統 RAG vs. LLM Wiki

面向傳統 RAG
（NotebookLM / 上傳檔案問答）		LLM Wiki
知識儲存位置原始文件的向量索引一份會被持續改寫的 markdown 檔案集
每次查詢做的事重新檢索 chunk → 重新拼裝答案先讀 index，直接讀整理好的頁面
跨文件綜合每次都要重新推理一次綜合結果已寫成頁面，之後直接讀
矛盾 / 過時資料不會主動標記ingest / lint 時主動比對、標記推翻
累積性無（stateless，問完即丟）有（compounding，越用越豐富）

三層架構

① Raw Sources（不可變）
原始文件——文章、論文、圖片、資料檔。LLM 只讀不改，是唯一的 source of truth。
② The Wiki（LLM 生成）
一堆互相連結的 markdown：摘要頁、實體頁、比較頁、synthesis 頁。LLM 完全擁有這層——建頁、更新頁、維護交叉引用。人讀，LLM 寫。
③ The Schema（配置檔）
CLAUDE.md / AGENTS.md：定義 wiki 結構、慣例、工作流程。讓 LLM 從「泛用聊天機器人」變成「有紀律的維基管理員」。

閉環關係：Schema 決定規則 → LLM 依規則把 Raw Sources 蒸餾進 Wiki → 人在 Wiki 層探索、發現規則不夠用時回頭修 Schema。這其實把「prompt engineering」升級成一份活的、持久化的 agent 治理文件。

三個核心操作

Ingest · 收錄

丟一份新來源 → LLM 讀 → 跟你討論重點 → 寫一頁摘要 → 更新 index → 回頭改相關的既有實體/概念頁（一份來源常動到 10–15 頁）→ log.md 補一筆。工程上最貴、最容易出錯的操作。

Query · 查詢

先讀 index 找相關頁 → 讀頁 → 綜合答案並附引用，可輸出成 markdown 頁 / 比較表 / 投影片 / 圖表。關鍵洞見：好答案要「歸檔回 wiki」變成新頁——臨時問出來的分析不該只活在對話紀錄裡。這把「查詢」和「寫作」變成同一個動作。

Lint · 健檢

定期讓 LLM 檢查：頁面矛盾、被推翻卻沒更新的舊主張、沒有 inbound link 的孤兒頁、被提到卻無獨立頁的重要概念、缺失的交叉引用、可用網搜補的缺口。三個操作裡最容易被跳過、但社群公認最不能省的一步。

實作細節（都不需要資料庫）

元件做什麼
index.md內容導向的目錄：每頁一行摘要 + 分類。查詢時先讀它找頁。Karpathy 明講 ~100 來源/數百頁內夠用，不需要 embedding RAG 基礎設施。
log.mdappend-only 時間軸，固定前綴 ## [2026-04-02] ingest | 標題，純 unix 工具可 parse（grep "^## \[" log.md | tail -5）。
Gitwiki 就是一個 git repo，版本歷史 / branching / 協作全部免費——這是選 markdown 而非資料庫的核心理由。
Obsidian當「IDE」：graph view 看拓撲（誰是 hub、誰是孤兒）、Web Clipper 把網頁轉 markdown、Dataview 對 frontmatter 查詢、Marp 直接產投影片。
qmd（選用）規模變大後接上：本地 BM25 + vector hybrid search + LLM re-rank，CLI/MCP 皆可。作者是 Shopify CEO Tobi Lütke，GitHub 近 28k stars，真實熱門工具。

工程師觀點

👍 好在哪

  • 用寫入放大換查詢品質：跨文件關係推理做一次、寫死進頁面，之後查詢是讀取。這是資料庫設計的經典 trade-off，只是換成用 LLM 當 ETL。
  • 零基礎設施起步：markdown + git + grep 就能跑，不用 vector DB、不用 embedding pipeline。對單人 PKM / 小型研究，摩擦力極低。
  • 人機分工用了三十年的隱喻：人做判斷、LLM 做記帳。用「維基管理員」這個直覺概念，讓非工程背景的人也能理解「我在跟一個永遠不會累的助手合作」。

⚠️ 取捨與隱憂（多數來自社群生產環境回饋，非原文提及）

隱憂具體表現緩解方式
偏誤 / 幻覺累積幻覺寫進 wiki 後，下次查詢讀到「被自己污染過的二手資料」，錯誤被當既定事實複合，比單次幻覺更難發現。頁面標「verify against source」逼 agent 對照原文——但快取成本翻倍。有實測：per-entity 頁反而比原檔還大，改用一張密集表格更有效率。
Drift（漂移）
最大殺手		ingest 時 under-update 交叉引用，頁面悄悄過時；自由格式連結幾年內就腐化。把 lint 從「選用」變強制排程；用 typed entity + 明確 schema 取代自由文字連結。
實體去重「新資訊該併入既有實體頁，還是開新頁」是 write-time 最難的判斷，易產生重複頁或誤合併。尚無公認解法，目前最開放的問題。
多人協作撞車沒有 authorship 邊界和 review gate，共享 wiki 會腐爛。生產案例採 PR review、每層一個 writer、來源用 file@SHA:lines 版本釘選而非複製全文。
規模上限index.md 過數百頁後需另搭 qmd，等於還是要碰基礎設施。用版本釘選路徑代替複製原文、用蒸餾版取代原始雜訊格式。
KP1 · 本質是 write-time ETL
把跨文件推理攤到寫入，查詢變讀取。用寫入放大換查詢品質與延遲。
KP2 · Query = 寫作
好答案要歸檔回 wiki 變新頁，這是跟 RAG 最大的行為差異。
KP3 · Lint 不能省
Karpathy 列為選用，但社群公認最不該跳過——不 lint，wiki 就漂移腐化。
KP4 · 解的是意志力，不是技術
它處理的是人類維護知識庫半途而廢的問題，不是檢索演算法問題。

什麼時候用、什麼時候別用

適合：個人研究型 PKM、長期主題追蹤（產業/公司深度研究）、閱讀陪伴。這類場景「查詢頻率遠高於寫入頻率、且容忍少量漂移」，正是甜蜜點。（本站自己在跑的股票 wiki 就是同一個模式的實例。）

不適合：需要嚴格事實可追溯、監管合規（金融/醫療決策依據）不能容忍 LLM 摘要層介入原文與結論之間——除非搭版本釘選引用；純一次性、不會重複查詢的任務也不值得付 ingest 的寫入成本。

跟既有方案怎麼選

方案何時選它而非 LLM Wiki
傳統 RAG / Vector DB資料變動快、不會重複查同一批知識、或資料量大到 wiki 化的寫入成本划不來。
GraphRAG（微軟）要的是「自動化廣度」而非「人指導的深度」；不在意可讀性、只要檢索品質。
Obsidian + Dataview（純手動）資料量小、你就想自己整理、不信任 LLM 寫入知識庫。
NotebookLM / 檔案上傳臨時性任務、不打算長期累積、怕 wiki 漂移風險。
qmd不是替代品，是互補——wiki 超過 index.md 手動維護規模（數百頁）時接上當搜尋層。
