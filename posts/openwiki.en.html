OpenWiki · Let an LLM Agent Auto-Write Your Codebase Wiki

A CLI from the LangChain team: it scans a repo, runs an agent loop to produce structured markdown docs (architecture, workflows, module map), and can run on CI to open a PR daily keeping them fresh.
It's not a docs tool for humans — it's about giving downstream AI coding agents (Claude Code / Cursor) a stable "repo manual" to consult.
Source: langchain-ai/openwiki · GitHub ↗ Type: CLI tool / MIT Compiled: 2026-07-16

TL;DR

🛠 OpenWiki turns "codebase docs" into an agent's output — scan the repo, generate openwiki/ markdown, run on CI to keep it updated, so coding agents have stable context.

Two modes: code mode generates docs for the current repo and maintains a marked block inside AGENTS.md/CLAUDE.md; personal mode connects Gmail/Notion/X and other connectors into a personal knowledge base. Under the hood it runs LangChain's own deepagents agent framework.

★ Very young (created 2026-06-22, exploded to 11.6k stars in ~3 weeks), still 0.x, behavior may change a lot. Better to study its design than to run it in production today.

The problem it solves

What AI coding agents lack most is stable, non-live repo context: every time they enter a large repo they re-discover the architecture from scratch. The traditional fix — hand-written CLAUDE.md / ADRs — depends entirely on human discipline to maintain.

OpenWiki's bet: if the docs are for agents to read, then let an agent write and maintain them — humans only review.

It runs an agent over the repo, producing a set of structured docs under openwiki/ (quickstart, architecture, cli, agent, operations, integrations…), and injects a "read openwiki/ first" hint into the root AGENTS.md/CLAUDE.md using <!-- OPENWIKI:START -->…END --> markers — rewriting only that block, leaving your other content untouched.

Two modes

code mode (default)
Generates openwiki/ docs for the current repo + maintains the marked block in AGENTS.md/CLAUDE.md. Before running, collects a change summary via git status/log/diff to feed the agent.
personal mode
Connects six connectors — git-repo, X, Notion, Gmail, Web Search (Tavily), Hacker News — ingesting fetched data into ~/.openwiki/wiki/ as a personal knowledge base; each connector can have multiple instances on separate schedules.

How it works

▸ Workflow: from repo scan to a daily auto-PR — skipped when nothing changed
PieceWhat it does
Agent skeletonCalls LangChain's own deepagents (createDeepAgent + CompositeBackend + FilesystemBackend), running on a custom OpenWikiLocalShellBackend whose virtual filesystem root maps to the target repo.
docs-only write guardIn code mode the backend blocks any write/edit outside openwiki/ (docs-only-backend.ts). If the agent tries to touch source, it's rejected at the code layer with an error — not left to prompt discipline.
Git evidenceThe host process runs git first to build a change summary: init grabs the last 20 commits, update the diff since the last recorded HEAD. Gives the model stable, non-live context.
Idempotent dedupAfter a run it takes a SHA-256 snapshot of openwiki/; if nothing changed it skips writing metadata — avoiding meaningless empty PRs on scheduled CI.
Multi-providercreateModel branches by provider (Anthropic / OpenAI / OpenRouter / Bedrock / Vertex / OpenAI-compatible), all on LangChain retry. Adding a provider = edit one file + one case.
State persistence@langchain/langgraph-checkpoint-sqlite stores conversation threads in ~/.openwiki/openwiki.sqlite, keyed by a hash of the repo path, so interactive mode can resume.

Key features

  • openwiki --init / --update one-shot generate/incrementally update repo docs; --print for one-off CI output.
  • Auto-maintains the marked block in AGENTS.md/CLAUDE.md so coding agents know to read the wiki first.
  • Built-in GitHub Actions / GitLab CI / Bitbucket Pipelines templates; a daily cron opens a PR to update docs by default.
  • Interactive Ink (React for terminal) UI: ask follow-ups live, swap keys with /api-key, /langsmith-key, LangSmith tracing supported.
  • Cross-platform OAuth (with an ngrok tunnel for callbacks), macOS LaunchAgents to schedule local connectors.
  • Supports OpenAI ChatGPT-subscription login (via a Codex backend, no API quota) and multiple billing models.

Quick start

npm install -g openwikiGlobal install (on Windows prefer npm/pnpm to avoid bun compiling better-sqlite3).
openwiki --initInteractive: pick provider / API key / model, generate openwiki/ docs.
openwiki --updateIncrementally update docs from the git diff.
openwiki -p "summarize what you can do"One-off, non-interactive, prints and exits.
openwiki personal --initPersonal knowledge-base mode.
openwiki auth notion|gmail|xAuthorize a connector via browser OAuth.

CI schedule: copy examples/openwiki-update.yml into .github/workflows/, set OPENROUTER_API_KEY (template defaults to OpenRouter + z-ai/glm-5.2), and it opens a PR daily.

An engineer's take

👍 Worth learning

  • docs-only write guard: blocking non-doc writes at the backend layer instead of via prompt — a great example of "don't outsource your security boundary to LLM self-discipline." Anyone doing "agent + file writes" should copy this.
  • Content snapshot dedup: SHA-256 comparison avoids meaningless empty PRs — an idempotent-CI detail many agent tools miss.
  • Centralized provider abstraction: config converges in constants.ts + one createModel branch, so adding a provider is cheap.
  • Dogfooding: the repo's own openwiki/ was generated by OpenWiki, reads quite well, has source maps — the strongest maturity signal.

When to use it, when not

Good fit: mid-to-large, frequently changing, multi-person repos with downstream AI coding agents, where you want agents to have a stable "repo manual."

Poor fit: small stable projects (maintenance cost > benefit), strict compliance on LLM output (internal finance/medical docs), or when you won't send API keys/repo content to a third-party provider. If you just want a project brief for Claude Code, claude /init is enough — no need for the whole connector/scheduling system.
