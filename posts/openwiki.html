OpenWiki · 讓 LLM agent 自動幫 codebase 寫維基

LangChain 官方團隊做的 CLI：掃描 repo，跑一個 agent loop 產出結構化 markdown 文件（架構、workflow、模組地圖），還能掛 CI 每天自動開 PR 更新。
定位不是給人看的文件工具——是讓下游的 AI coding agent（Claude Code / Cursor）有穩定「repo 說明書」可查。
來源:langchain-ai/openwiki · GitHub ↗ 性質:CLI 工具 / MIT 整理日:2026-07-16

TL;DR

🛠 OpenWiki 把「codebase 的文件」變成 agent 跑出來的產物——掃 repo、生成 openwiki/ markdown、掛 CI 持續更新，讓 coding agent 有穩定上下文可查。

兩種模式：code 模式對當前 repo 生成文件並維護 AGENTS.md/CLAUDE.md 的專屬區塊；personal 模式接 Gmail/Notion/X 等 connector，做個人知識庫。底層跑 LangChain 自家的 deepagents agent 框架。

★ 極年輕（2026-06-22 建立、約 3 週爆紅到 11.6k star），版本還在 0.x，行為仍可能大改。看它學設計比直接上生產更划算。

它解決什麼問題

AI coding agent 最缺的是穩定、非即時查詢的 repo 上下文：每次進到一個大 repo 都要重新摸索架構。傳統做法是人手寫 CLAUDE.md / ADR，但維護成本全靠人的紀律。

OpenWiki 的賭注：既然文件是給 agent 讀的，那就讓 agent 自己寫、自己維護——人只需要 review。

它跑 agent 掃 repo，產出 openwiki/ 目錄下一組結構化文件（quickstart、architecture、cli、agent、operations、integrations…），並在 repo 根目錄的 AGENTS.md/CLAUDE.md 用 <!-- OPENWIKI:START -->…END --> 標記塞一段「請先讀 openwiki/ 再工作」的提示——只重寫這段，不動你其他內容。

兩種模式

code 模式（預設）
對當前 repo 產出 openwiki/ 文件 + 維護 AGENTS.md/CLAUDE.md 標記區塊。跑前先用 git status/log/diff 收集變更摘要餵給 agent。
personal 模式
接 git-repo、X、Notion、Gmail、Web Search（Tavily）、Hacker News 六種 connector，把抓下來的資料 ingest 進 ~/.openwiki/wiki/，變成個人知識庫；同一 connector 可設多個 instance 分開排程。

運作原理

▸ 運作流程：從 repo 掃描到每日自動開 PR，內容沒變就跳過
環節做什麼
Agent 骨架呼叫 LangChain 自家 deepagents（createDeepAgent + CompositeBackend + FilesystemBackend），跑在自訂的 OpenWikiLocalShellBackend 上，虛擬檔案系統根目錄對到目標 repo。
docs-only 寫入護欄code 模式下 backend 直接擋掉「非 openwiki/ 路徑」的 write/edit（docs-only-backend.ts）。agent 想改原始碼會被程式碼層拒絕並回錯誤——不是靠 prompt 約束。
Git 證據收集host process 先跑 git 指令產生變更摘要：init 抓最近 20 筆 commit，update 抓上次 HEAD 之後的 diff。給模型穩定、非即時查詢的上下文。
冪等去重完成後對 openwiki/ 算 SHA-256 snapshot，內容沒變就不寫 metadata——避免排程 CI 產生無意義的空 PR。
多 provider 抽象createModel 依 provider 分支建 chat model（Anthropic / OpenAI / OpenRouter / Bedrock / Vertex / OpenAI-compatible），全走 LangChain retry。加新 provider ＝ 改一個檔案 + 加一個 case。
State 持久化@langchain/langgraph-checkpoint-sqlite 把對話 thread 存到 ~/.openwiki/openwiki.sqlite，key 是 repo 路徑 hash，互動模式可接續對話。

主要功能

  • openwiki --init / --update 一鍵生成/增量更新 repo 文件，--print 給 CI 用的一次性輸出。
  • 自動維護 AGENTS.md/CLAUDE.md 專屬區塊，讓 coding agent 自動知道先讀 wiki。
  • 內建 GitHub Actions / GitLab CI / Bitbucket Pipelines 範本，預設每天 cron 自動開 PR 更新文件。
  • 互動式 Ink（React for terminal）UI，可即時追問、/api-key、/langsmith-key 動態換 key，支援 LangSmith tracing。
  • 跨平台 OAuth（含 ngrok 隧道給 callback）、macOS LaunchAgents 排程本地 connector。
  • 支援 OpenAI ChatGPT 訂閱登入（走 Codex backend，不吃 API 額度）等多種計費模型。

快速上手

npm install -g openwiki全域安裝（Windows 建議走 npm/pnpm，避免 bun 需編譯 better-sqlite3）。
openwiki --init互動式：選 provider / API key / model，生成 openwiki/ 文件。
openwiki --update依 git diff 增量更新文件。
openwiki -p "摘要一下你能做什麼"一次性、非互動、印完就結束。
openwiki personal --init個人知識庫模式。
openwiki auth notion|gmail|x走瀏覽器 OAuth 授權 connector。

CI 排程：把 examples/openwiki-update.yml 複製到 .github/workflows/，設好 OPENROUTER_API_KEY（範本預設 OpenRouter + z-ai/glm-5.2）即可每天自動開 PR。

工程師觀點

👍 值得學的設計

  • docs-only 寫入護欄：用 backend override 從程式碼層擋掉 agent 亂寫非文件路徑，而非單靠 prompt——「別把安全邊界外包給 LLM 自律」的好示範，做「agent + 檔案寫入」的人都該參考。
  • 內容 snapshot 去重：SHA-256 比對避免無意義空 PR，這種 idempotent CI job 細節很多 agent 工具會漏。
  • Provider 抽象集中化：設定收斂在 constants.ts + createModel 一處，加 provider 成本低。
  • Dogfooding：repo 裡的 openwiki/ 就是 OpenWiki 自己生成的，讀起來品質相當高、有 source map——最有力的成熟度佐證。
▸ docs-only 護欄：寫入邊界擋在 backend，不外包給 LLM 自律

⚠️ 取捨與隱憂

隱憂具體表現怎麼看
極年輕，數字打折2026-06-22 建立、首個 tag 0.0.1 在 07-05、現 0.1.2。11.6k star 是被 Trendshift/HN 推上熱榜的 3 週爆紅曲線。版本 0.x，行為仍可能大改；star ≠ 生產驗證。110 個 open issues 代表還在快速填坑。
強綁 langchain 生態底層吃自家 deepagents（1.x 初期）、@langchain/* 全鎖近期 major。依賴鏈都很新，任一環破壞性升級都會波及；跟非 LangGraph 框架整合成本高。
沒有幻覺文件驗證本質是「把 repo 塞進 context 讓 LLM 總結」，超大 monorepo 或高度動態程式碼準確度會掉。沒看到生成後 lint 檢查（例如驗證文件引用的檔案路徑是否存在）。
personal 模式隱私面要讀 Gmail/X/Notion 私人資料丟給 LLM 摘要，OAuth token 存 ~/.openwiki/.env 明文。企業採用前自行評估資料外流，尤其切到非受信任 provider（自架 gateway）時。
CI 每天多一個 PR自動開的文件 PR 仍要人 review。沒設好 auto-merge 規則反而增加管理負擔。

什麼時候用、什麼時候別用

適合：中大型、變動頻繁、多人協作、且下游有用 AI coding agent 的 repo，想讓 agent 有穩定「repo 說明書」可查。

不適合：小型穩定專案（維護成本 > 效益）、對 LLM 產出有嚴格合規要求（金融/醫療內部文件）、或不想把 API key/repo 內容送第三方 provider 的場景。若只是想要一份給 Claude Code 用的專案說明，claude /init 就夠，不必引入整套 connector/排程系統。

類似專案怎麼選

專案何時選它而非 OpenWiki
DeepWiki（Devin/Cognition）想「馬上看別人公開 repo 的文件」——免安裝、免自帶 API key、線上瀏覽。OpenWiki 反過來是 CLI-first、可自架、可持續整合進自己 CI。
Repomix只想把 repo 打包成單一 LLM 友善文字檔自己餵給 ChatGPT/Claude，不需要 agent loop 與自動產文件流程時，更輕量。
Sourcegraph Cody要的是「即時查 codebase」（code search + RAG 問答）而非「離線可讀的靜態 wiki」。
Claude Code /init只要一份給 Claude Code 的專案說明，一次性掃描即可，比整套 connector/排程輕得多。只有需要「跨多來源持續自動更新」時 OpenWiki 才划算。
手寫 ADR + MkDocs團隊本來就有紀律做架構決策紀錄、要求 100% 準確度、不信任 LLM 自動生成層。
