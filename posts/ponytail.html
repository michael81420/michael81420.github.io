Ponytail · 讓 AI agent 像最懶的資深工程師那樣寫 code

一個跨平台的「行為規則注入套件」——不執行任何程式，只在 AI agent 動筆前塞進一句話：「這段 code 到底需不需要存在？」
口號：The best code is the code you never wrote.
來源:GitHub · DietrichGebert/ponytail ↗ 主語言:JavaScript / Python 授權:MIT 整理日:2026-06-26

TL;DR

🪶 Ponytail 是 prompt 層的「過度工程 linter」，不是框架、不是 runtime。

它把一組「資深工程師的偷懶直覺」寫成規則，透過各平台的 hook / config 機制，注入到 Claude Code、Cursor、Codex、Copilot 等 14+ 個 AI agent 的 system context。核心是一條決策梯：寫任何一行 code 前，由上往下問，碰到第一個成立的就停手。

★ 效果完全取決於模型的「指令遵循能力」——沒有 guardrail 強制驗證 agent 真的走過這條梯子。買的是思考框架，不是保證數字。

它解決什麼問題

AI agent 寫 code 最常見的毛病是 hallucinated bloat（幻覺式臃腫）：

  • 幫你裝整包 lodash，只為了寫 x => x * 2
  • 幫你建一個 DatePickerWrapperComponent，包住一個原生 <input type="date">
  • 幫單一用途的 code 蓋一層 interface + factory + config

Ponytail 直接在認知層堵住這個洞——不是事後 code review 補救，而是在 agent 開始想之前就改變它的預設傾向。

核心：決策梯（Decision Ladder）

agent 在寫 code 前必須由上往下走過這個梯子，碰到第一個適用點就停：

1 · 這根本不需要存在嗎？
直接跳過（YAGNI）。投機性需求一律砍。
2 · Codebase 已有實作？
重用，不要重寫。
3 · 標準函式庫有？
用 stdlib。
4 · 原生平台 feature？
用原生，如 <input type="date"> 而非 picker library。
5 · 已安裝的 dependency？
用它，別為幾行 code 加新依賴。
6 · 一行搞定？
寫一行。都不行才寫最小可行實作。

★ 不可跳過的安全底線（所有強度等級都保留）：trust boundary 驗證、資料遺失防護、security、accessibility、防 data loss 的 error handling。明確區分「可以偷懶的」和「絕對不能省的」，是這個工具的工程誠信所在。

它怎麼運作

Ponytail 不是 standalone 工具，而是跨平台的規則同步 + 注入系統，依各 agent 的能力分三種注入方式：

機制對象做法
靜態 rulesCursor / Windsurf / Cline / Copilot在 repo 放 .cursor/rules/、.clinerules、.github/copilot-instructions.md，啟動時讀入為 always-on context
生命週期 HookClaude Code / Codex（CLI）用 Node.js lifecycle hooks 在 session 開始 / 每次 prompt 時動態注入
Skill 命令Claude Code / Gemini CLI 等提供可呼叫指令（見下）

scripts/sync-rules.js 負責讓「核心規則文字」在所有平台格式間保持一致——這是維護 14+ agent 支援的關鍵基礎設施。

可呼叫的指令：

  • /ponytail [lite|full|ultra|off] — 切換強度
  • /ponytail-review — 分析目前 diff 是否過度工程
  • /ponytail-audit — 掃整個 repo 的複雜度
  • /ponytail-debt — 把延遲優化的 comment 整理成可執行 ledger
  • /ponytail-gain — 顯示 benchmark 數據

強度等級

等級行為
lite溫和建議，不強制
full預設；平衡的最小化
ultra對嚴重過度工程的 codebase 激進最小化
off關閉

預設值可用環境變數 PONYTAIL_DEFAULT_MODE 或 ~/.config/ponytail/config.json 設定。

工程師觀點

👍 好在哪

  • 問題診斷精準：在認知層堵 bloat，而非事後補救。
  • 零 runtime 開銷：純 prompt 層，對正式環境毫無影響。
  • 跨平台：IDE 與 CLI 兩型都覆蓋，sync 腳本讓規則維護不失控。
  • 有安全底線：明確列出「絕對不能省」，避免淪為讓 agent 略過安全檢查的工具。

⚠️ 取捨與隱憂

  • 完全依賴 LLM 的指令遵循：沒有 guardrail 驗證 agent 真的走過決策梯。context 滿時規則可能被模型忽略。
  • Benchmark 需存疑：README 宣稱「54% less code、27% faster、100% safe」是自述、在特定 12 個任務上測的，缺第三方驗證；已精簡的 codebase 效益趨近零。
  • 「懶」不等於「對」：YAGNI 那步由 agent 自己判斷，可能誤砍真正必要的邏輯——ultra 模式在 legacy code 上尤其危險。
  • 維護負擔：14+ 平台 = 14+ 套格式要同步，平台改規格時有滯後風險。
KP1 · 本質是 prompt 層 linter
不執行程式，只插入「思考前先問的清單」。輕量無風險，但效果取決於模型。
KP2 · 決策梯是精華
YAGNI → 原生 → stdlib → 已有依賴 → 一行 → MVP，把資深直覺明確寫給 agent。
KP3 · 數字當方向，非保證
54% less code 在過度工程場景成立，已精簡 codebase 近乎零效益。
KP4 · 安全底線是誠信標誌
保留不可省略清單，避免變成「偷懶略過安全」的工具。

什麼時候用、什麼時候別用

適合：agentic workflow 下 agent 常「多做事」、想控 LLM 成本、legacy refactor 要聚焦最小改動、成熟 codebase 不想引入新依賴。

不適合：greenfield 需要搭完整鷹架、prototype / spike 要創作空間、ultra + 複雜業務邏輯（過度刪減風險高）、不熟 YAGNI 的初級工程師寫第一個專案。
