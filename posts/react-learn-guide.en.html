TL;DR

🧠 One-line mental model: UI = f(state).

You never touch the DOM directly. You change state; React re-runs your component, diffs the result, and updates the DOM. The entire framework exists to serve this one rule.

The official path has four stages, and finishing them is enough to ship production code: ① Describing the UI → ② Adding Interactivity → ③ Managing State → ④ Escape Hatches.

★ 90% of React bugs come from two things you didn't think through: which layer the state belongs on, and when NOT to use useEffect. Every chapter below gives you a decision tree you can follow.

0 · What React actually solves, and when to reach for it

What React is: a library for building UIs out of components. Each component is a JS function that returns some UI, can hold its own state, and can be reused. You only describe "what the UI looks like for each state"; React handles all the DOM work.

✅ Reach for React when
the UI changes often with data/interaction, there's a lot of interdependent state, and UI needs reuse (admin panels, SaaS, dashboards, highly interactive front-ends).
🤔 You don't need React when
it's a static content page with one or two small interactions — plain HTML/CSS/JS is enough (like the site you're reading 😉). A framework there is just added dependency.

The rule in one line: the complexity of your interactive state decides whether you need React. The more state you have and the more it interacts, the more manual DOM syncing React saves you.

1 · Describing the UI — what the screen looks like

1.1 Components & JSX: the UI is a function that returns markup

A component is a capitalized JS function that returns JSX. JSX lets you write HTML-like markup inside JS, and curly braces { } drop in any JS value or expression.

function Welcome() {
  const name = "Michael";
  return <h1>Hi, {name}!</h1>;   // { } holds JS
}
// Usage: nested like an HTML tag by another component
<Welcome />

When to extract a new component: a chunk of JSX repeats, or a component grows too big to read at a glance / does too many things. Don't over-split up front — extract when it repeats or gets complex.

1.2 Props: read-only data from parent to child (one-way, top-down)

Props are the arguments a parent passes to a child, and the child can't change them. This is half of React's one-way data flow: data always flows down.

function Avatar({ name, size }) {   // destructure props
  return <img alt={name} width={size} />;
}
<Avatar name="Michael" size={48} />   // parent sets the values

When to use: whenever a child needs to display or customize the parent's data. Need to pass something "up"? The parent passes a function as a prop and the child calls it (see 3.3, lifting state).

1.3 Conditional rendering & lists: why key must not be the array index

// Conditions: use && or a ternary
{isLoggedIn && <Dashboard />}
{isLoggedIn ? <Dashboard /> : <Login />}

// Lists: map, and give each a stable unique key
{users.map(u => <Row key={u.id} user={u} />)}   // ✅ data id
{users.map((u, i) => <Row key={i} user={u} />)}   // ❌ index

What a key is: React uses it to recognize "which item is the same one across re-renders." Why not the index: once a list is reordered, inserted into, or filtered, an item's index shifts, and React attaches UI state (half-typed input, checkbox state) to the wrong item.

List situationWhat to use for key
Data has a unique idUse the id (best)
No id but never reordersIndex is tolerable
Sorted / inserted / filteredNever use the index

1.4 Keeping components pure

During render, a component should compute its UI from props and state only — not mutate outside variables, not make network requests (leave those to events or Effects).

Why it matters: a pure function gives the same output for the same input → predictable, cacheable, safe for React to re-run. Strict Mode deliberately calls each component twice in development to catch impure ones; a pure component runs twice with identical results and doesn't break.

2 · Adding Interactivity — making the UI move

2.1 Events: pass the function itself, not the result of calling it

<button onClick={handleClick}>Click</button>    // ✅ pass the function
<button onClick={handleClick()}>Click</button>  // ❌ called during render

2.2 useState: a component's "memory"

For the UI to change, you need a value that "survives across re-renders" — that's state. A normal variable won't do: it resets on every render, and changing it won't trigger a UI update.

const [count, setCount] = useState(0);
//     ↑current  ↑setter  ↑initial value
function handleClick() { setCount(count + 1); }

setCount does two things: remember the new value, and schedule a re-render. This button works exactly that way:

2.3 Render & commit: the three steps of a UI update

Key point: "render" is just React calling your function to produce a description of what the UI should look like — it hasn't touched the browser yet. Only at "commit" does React touch the real DOM, and only the parts that changed. That's why you never call document.getElementById yourself.

2.4 ★ State is a "snapshot", plus batching

The most counter-intuitive part, and the one people trip on most: within a single event, a state variable is a fixed snapshot. setCount does not change the count on the current line — it only registers "use the new value on the next render."

// count is currently 0
setCount(count + 1); // queue: use 0+1
setCount(count + 1); // count is still the snapshot 0 → queue 0+1
setCount(count + 1); // same 0+1 → net result is only +1!

// Want to accumulate? Use an updater — it gets the previous result
setCount(c => c + 1); // 0→1
setCount(c => c + 1); // 1→2
setCount(c => c + 1); // 2→3  ✅ actually +3

Click and feel the difference — same "call it three times", very different result:

</> Expand: the full React component behind this demo 
function BatchDemo() {
  const [count, setCount] = useState(0);
  return (<>
    <span>{count}</span>
    <button onClick={() => {
      setCount(count + 1); // same snapshot all 3 times
      setCount(count + 1);
      setCount(count + 1); // → only +1
    }}>count+1 × 3</button>
    <button onClick={() => {
      setCount(c => c + 1); // gets the previous result
      setCount(c => c + 1);
      setCount(c => c + 1); // → +3
    }}>c => c+1 × 3</button>
  </>);
}

A batch of setStates is batched by React into a single render, for performance. Whether you want to accumulate (updater) or overwrite (pass a value) is your call — pick right and you won't get bitten.

2.5 Updating object / array state: immutably

Don't mutate object or array state in place — produce a new one and pass it in.

// ❌ mutate in place: React can't detect it, UI won't update
user.age = 30; setUser(user);
list.push(item); setList(list);

// ✅ produce a new object / array
setUser({ ...user, age: 30 });
setList([...list, item]);

Why: React decides whether to re-render by checking if the new and old state are the same reference. Mutate in place and the reference doesn't change, so React thinks nothing happened and the UI stays put.

3 · Managing State — which layer, how to organize

This chapter is React's real difficulty, and where judgment separates people. Here's the master map first: which layer a piece of state belongs on — just follow the tree.

3.1 Reflecting input with state (controlled components)

React's signature idea: the UI is a function of state. An input doesn't keep its own value; its value is bound to state and onChange updates state. Try typing:

function Mirror() {
  const [text, setText] = useState("");
  return (<>
    <input value={text} onChange={e => setText(e.target.value)} />
    <span>{text.toUpperCase()}</span>   // derived value, computed inline
  </>);
}

3.2 Choosing the state structure (three rules)

  • Don't store what you can compute: fullName comes from first + last — compute it during render, don't keep a separate state (it'll go out of sync).
  • Don't duplicate: keep one source of truth for each piece of data; reference it elsewhere.
  • Keep it flat: deeply nested state is painful to update; flatten it.

3.3 Lifting state up

When two sibling components need to share the same data, move the state to their closest common parent. The parent passes the value down via props and receives changes back via a callback. The slider below is the parent's state; two children read it in sync:

</> Expand: the full React component behind this demo 
function Thermo() {
  const [temp, setTemp] = useState(25); // state lifted to the parent
  return (<>
    <input type="range" value={temp}
           onChange={e => setTemp(+e.target.value)} />
    <Celsius value={temp} />     // both children read
    <Fahrenheit value={temp} />  // the same source
  </>);
}
const Celsius = ({ value }) => <div>{value}°C</div>;
const Fahrenheit = ({ value }) => <div>{Math.round(value*9/5+32)}°F</div>;

Once lifted, the two children are always consistent — because they read the same source, not private copies.

3.4 Preserving and resetting state

State is tied to a component's position in the UI tree. Same position, same component type → React preserves the state; change the key → React treats it as a new component and resets the state.

// Want to clear the form when switching users? Change the key.
<ProfileForm key={userId} />

3.5 useReducer: centralize complex update logic

When one piece of state has many kinds of updates scattered across handlers, switch it to a reducer: all the "how it changes" lives in one function, and the component just dispatches an action.

function reducer(state, action) {
  switch (action.type) {
    case "inc":   return { count: state.count + 1 };
    case "dec":   return { count: state.count - 1 };
    case "reset": return { count: 0 };
    default:      return state;
  }
}
function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 });
  return (<>
    <span>{state.count}</span>
    <button onClick={() => dispatch({ type: "inc" })}>+1</button>
    <button onClick={() => dispatch({ type: "dec" })}>−1</button>
    <button onClick={() => dispatch({ type: "reset" })}>reset</button>
  </>);
}

3.6 Context: pass the same data through many layers

When many layers and many components need the same "app-wide" data (theme, current user, locale), threading it through props is painful (called prop drilling). Context lets you provide it once at a top-level Provider, and any component below — at any depth — grabs it with useContext.

When to use: data that's "shared app-wide config." Don't overuse it — not every piece of state belongs in Context, or a single change re-renders a huge subtree. Most state is fine as local + lifted.

3.7 Reducer + Context = lightweight global state

Put a useReducer's state/dispatch into Context and you get a lightweight global-state solution with zero libraries: centralized logic (reducer), broad reach (context). Most small-to-mid apps are fine with this combo — no need to rush to Redux/Zustand.

4 · Escape Hatches — when you need to step outside React

"Escape hatches" are the tools for when you must touch the world outside React (the DOM, timers, network, third-party libs). They're powerful — and the easiest to misuse, especially useEffect.

4.1 refs: hold a value that doesn't affect the UI / grab a DOM node

useRef gives you a box that persists across re-renders but changing it doesn't trigger a render. Use it for things that shouldn't affect the UI (a timer id, the previous value), or to grab a real DOM node (focus(), measure, play a video).

Below contrasts state and ref — notice the UI "doesn't move" when you click the ref button:

function RefVsState() {
  const [state, setState] = useState(0);
  const ref = useRef(0);
  console.log("render"); // only setState makes this line run again
  return (<>
    <p>state: {state} · ref: {ref.current}</p>
    <button onClick={() => setState(state + 1)}>setState + 1</button>
    <button onClick={() => { ref.current += 1; }}>ref + 1</button>
  </>);
}
const ref = useRef(0);
ref.current += 1;         // change it → no re-render
inputRef.current.focus(); // grab a DOM node, operate directly

4.2 useEffect: synchronize with an "external system"

An Effect is a side effect React runs after it commits to the DOM, specifically to sync with systems outside React: subscribing to events, starting/stopping timers, connecting a WebSocket, driving a non-React DOM library.

useEffect(() => {
  const id = setInterval(tick, 1000); // setup
  return () => clearInterval(id);        // cleanup (always clean up)
}, [/* deps: list every reactive value you use */]);

The dependency array must honestly list every prop/state used inside the Effect; the official ESLint rule catches omissions. Each time a dep changes, React cleans up the previous run, then sets up this one — so always write the cleanup (or timers and subscriptions pile up).

4.3 ★ You Might Not Need an Effect (the most important section)

The biggest beginner trap: stuffing everything into useEffect. The docs devote a whole page to "You Might Not Need an Effect." Figure out which kind of thing you're actually doing:

function NameForm() {
  const [first, setFirst] = useState("");
  const [last, setLast] = useState("");
  const full = (first + " " + last).trim(); // ✅ computed in render, no Effect
  return (<>
    <input value={first} onChange={e => setFirst(e.target.value)} />
    <input value={last}  onChange={e => setLast(e.target.value)} />
    <b>{full}</b>
  </>);
}
// ❌ deriving state via Effect: extra render, easy to desync
const [full, setFull] = useState("");
useEffect(() => { setFull(first + " " + last); }, [first, last]);

// ✅ just compute it during render
const full = first + " " + last;

4.4 Custom Hooks: extract stateful logic for reuse

When several components need "the same chunk of stateful/Effect logic" (say useOnlineStatus, useFetch), extract it into a useXxx function. A custom Hook is just a function that composes built-in Hooks, named with a use prefix so React recognizes it.

function useOnlineStatus() {
  const [online, setOnline] = useState(navigator.onLine);
  useEffect(() => {
    const on = () => setOnline(true), off = () => setOnline(false);
    window.addEventListener("online", on);
    window.addEventListener("offline", off);
    return () => { /* remember to remove listeners */ };
  }, []);
  return online;
}
