TL;DR

🧠 一句話心智模型：畫面 = f(state)。

你不直接改畫面（DOM），而是改 state（狀態）；React 幫你重新執行元件、算出畫面差異、再更新 DOM。整個框架都在服務這一條規則。

官方學習路線四個階段，學完就能寫 production：① 描述 UI → ② 加入互動 → ③ 管理 state → ④ 逃生艙。

★ 90% 的 React bug 只來自兩件事沒想清楚：state 該放在哪一層、以及什麼時候「不要」用 useEffect。這篇每一章都給你可以照著走的判斷樹。

0 · React 到底解決什麼、何時才該用

React 是什麼：一套用「元件（component）」組畫面的函式庫。每個元件是一個回傳畫面的 JS 函式，可以帶自己的狀態、可以重用。你只負責「描述每個 state 下畫面長怎樣」，剩下的 DOM 操作 React 全包。

✅ 適合用 React
畫面會隨資料/互動頻繁變化、狀態多且互相牽動、UI 要重用（後台、SaaS、儀表板、互動性強的前端 App）。
🤔 不必上 React
純靜態內容頁、只有一兩個小互動——原生 HTML/CSS/JS 就夠了（像你正在看的這個站 😉）。硬上框架只是增加依賴。

判斷準則一句話：互動狀態的複雜度決定要不要 React。狀態越多、越會互相影響，React 幫你省的手動 DOM 同步就越多。

1 · 描述 UI — 畫面長什麼樣

1.1 元件與 JSX：畫面就是回傳 markup 的函式

元件是大寫開頭、回傳 JSX 的 JS 函式。JSX 讓你在 JS 裡直接寫 HTML 般的標記，用大括號 { } 塞進任何 JS 值或表達式。

function Welcome() {
  const name = "Michael";
  return <h1>Hi, {name}!</h1>;   // { } 裡放 JS
}
// 用法：像 HTML 標籤一樣被別的元件嵌進去
<Welcome />

何時該拆成新元件：一段 JSX 重複出現、或一個元件長到「一眼看不完 / 同時管太多事」，就抽出來。不用一開始就過度切割——重複或變複雜時再拆。

1.2 Props：父傳子的唯讀資料（單向由上往下）

Props 是父元件傳給子元件的參數，子元件不能改它。這是 React 「資料單向流」的一半：資料永遠由上往下傳。

function Avatar({ name, size }) {   // 從 props 解構
  return <img alt={name} width={size} />;
}
<Avatar name="Michael" size={48} />   // 父層決定值

何時用：子元件需要顯示或客製化父層的資料時。想「往上傳」怎麼辦？父層傳一個 函式 當 prop 下去，子層呼叫它（見 3.3 提升 state）。

1.3 條件渲染 & 清單：key 為何不能用陣列 index

// 條件：用 && 或三元運算
{isLoggedIn && <Dashboard />}
{isLoggedIn ? <Dashboard /> : <Login />}

// 清單：map 出來，每個都要穩定唯一的 key
{users.map(u => <Row key={u.id} user={u} />)}   // ✅ 用資料 id
{users.map((u, i) => <Row key={i} user={u} />)}   // ❌ 用 index

key 是什麼：React 用 key 認出「重新 render 前後，哪個項目是同一個」。為什麼不能用 index：清單一旦重排、插入、刪除，同一筆資料的 index 會變，React 會把畫面狀態（例如輸入框裡打到一半的字、勾選狀態）綁到錯的項目上。

清單情況key 該用什麼
資料有唯一 id用 id（最佳）
沒有 id 但穩定不重排index 勉強可以
會排序 / 增刪 / 篩選絕對不要用 index

1.4 保持元件純淨（pure）

元件在 render 時應該只根據 props 和 state 算出畫面，不去改外部變數、不做網路請求之類的 side effect（那些留給事件或 Effect）。

為什麼重要：純函式 → 同樣輸入得同樣輸出 → 可預測、可快取、可被 React 安全地重跑。React 的嚴格模式（Strict Mode）故意在開發時把每個元件呼叫兩次來抓不純的元件，純的元件跑兩次結果一樣、不會爆。

2 · 加入互動 — 讓畫面會動

2.1 事件：傳「函式本身」，不是呼叫結果

<button onClick={handleClick}>按我</button>    // ✅ 傳函式
<button onClick={handleClick()}>按我</button>  // ❌ render 時就被呼叫了

2.2 useState：元件的「記憶」

畫面要會變，就需要一個「跨 re-render 還記得」的值，這就是 state。用一般變數不行——一般變數每次 render 都被重設，而且改它不會觸發畫面更新。

const [count, setCount] = useState(0);
//     ↑目前值   ↑改它的函式   ↑初始值
function handleClick() { setCount(count + 1); }

setCount 做兩件事：記住新值、並排一次 re-render。下面這顆按鈕就是這樣運作的：

互動：useState = 元件的記憶
0 按下去 → setCount → 元件函式重新執行 → 畫面更新。用普通變數的話，這個數字永遠是 0。
</> 展開：這個 demo 的 React 完整寫法 
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      count + 1 → {count}
    </button>
  );
}

2.3 render & commit：畫面更新的三步驟

重點：「渲染」只是 React 呼叫你的函式算出一份「畫面該長怎樣」的描述，還沒碰到瀏覽器。只有「提交」時 React 才動真實 DOM，而且只改有變的部分。這就是為什麼你不用自己 document.getElementById。

2.4 ★ state 是一張「快照」，加上批次更新

最反直覺、也最多人踩的一點：在同一次事件裡，state 變數是固定的快照。setCount 不會馬上改掉當下這行的 count，它只是登記「下一次 render 請用新值」。

// count 現在是 0
setCount(count + 1); // 排入：用 0+1
setCount(count + 1); // count 還是快照 0 → 又排入 0+1
setCount(count + 1); // 一樣 0+1 → 結果只 +1！

// 想連續加？用 updater 函式，拿到的是「前一次的結果」
setCount(c => c + 1); // 0→1
setCount(c => c + 1); // 1→2
setCount(c => c + 1); // 2→3  ✅ 真的 +3

親手點點看差別——同樣「呼叫三次」，結果差很多：

互動：快照 vs updater 函式（一次事件呼叫三次 setState）
0
先按左邊那顆——雖然呼叫三次，卻只 +1。
</> 展開：這個 demo 的 React 完整寫法 
function BatchDemo() {
  const [count, setCount] = useState(0);
  return (<>
    <span>{count}</span>
    <button onClick={() => {
      setCount(count + 1); // 三次讀同一張快照
      setCount(count + 1);
      setCount(count + 1); // → 只 +1
    }}>count+1 × 3</button>
    <button onClick={() => {
      setCount(c => c + 1); // 拿前一次結果
      setCount(c => c + 1);
      setCount(c => c + 1); // → +3
    }}>c => c+1 × 3</button>
  </>);
}

同一批 setState 會被 React 批次（batch）成一次 render，效能才好。你要的是累加（updater 函式）還是覆寫（直接給值），自己選對就不會出錯。

2.5 更新物件 / 陣列 state：不可變（immutable）

物件和陣列型的 state 不要就地改（mutate），要產生一個新的傳進去。

// ❌ 就地改，React 偵測不到、畫面不更新
user.age = 30; setUser(user);
list.push(item); setList(list);

// ✅ 產生新物件 / 新陣列
setUser({ ...user, age: 30 });
setList([...list, item]);

為什麼：React 靠比較「新舊 state 是不是同一個 reference」來決定要不要 re-render。你就地改，reference 沒變，React 以為沒事，畫面就不動。

你想做不可變寫法
加到尾端[...arr, x]
刪掉某筆arr.filter(a => a.id !== id)
改某筆arr.map(a => a.id===id ? {...a, ...patch} : a)
改物件某欄{ ...obj, key: value }

3 · 管理 state — 放哪一層、怎麼組織

這章是 React 的真正難點，也是判斷力的分水嶺。先給你一張總地圖：一個 state 該放在哪裡，照這棵樹走。

3.1 用 state 反映輸入（controlled component）

React 的招牌思路：畫面是 state 的函式。輸入框不自己記值，而是 value 綁 state、onChange 改 state。試著在下面打字：

互動：受控輸入 — 畫面永遠跟著 state
→ 即時大寫鏡像： —
</> 展開：這個 demo 的 React 完整寫法 
function Mirror() {
  const [text, setText] = useState("");
  return (<>
    <input value={text} onChange={e => setText(e.target.value)} />
    <span>{text.toUpperCase()}</span>   // 衍生值直接算
  </>);
}

3.2 選對 state 結構（三原則）

  • 能算出來的不要存：fullName 由 first + last 算，render 時算就好，別另開一個 state（會不同步）。
  • 不要重複：同一份資料只留一份真相，其他地方引用它。
  • 盡量扁平：深層巢狀的 state 更新起來很痛，攤平比較好改。

3.3 提升 state（lifting state up）

兩個兄弟元件要共用同一份資料時，把 state 移到它們最近的共同父層，父層用 props 把值傳下去、用 callback 把改動收上來。下面的滑桿是父層的 state，兩個子元件同步讀它：

互動：提升 state — 一份來源，兩個子元件同步
25°C
子元件 A · 攝氏
77°F
子元件 B · 華氏
</> 展開：這個 demo 的 React 完整寫法 
function Thermo() {
  const [temp, setTemp] = useState(25); // state 提升到父層
  return (<>
    <input type="range" value={temp}
           onChange={e => setTemp(+e.target.value)} />
    <Celsius value={temp} />     // 兩個子元件讀
    <Fahrenheit value={temp} />  // 同一份來源
  </>);
}
const Celsius = ({ value }) => <div>{value}°C</div>;
const Fahrenheit = ({ value }) => <div>{Math.round(value*9/5+32)}°F</div>;

state 提升後，兩個子元件永遠一致——因為它們讀的是同一個來源，不是各自複製一份。

3.4 保留與重置 state

state 綁在「元件在畫面樹裡的位置」。同一個位置、同一種元件 → React 保留 state；換掉 key → React 把它當成新元件、重置 state。

// 切換使用者時想清空表單？換 key 逼它重置
<ProfileForm key={userId} />

3.5 useReducer：把複雜的更新邏輯集中

當一個 state 有很多種更新動作、邏輯散在各個 handler 裡時，把它換成 reducer：所有「怎麼變」集中在一個函式，元件只負責 dispatch 一個 action。

互動：useReducer — 元件只 dispatch，邏輯集中在 reducer
0
</> 展開：這個 demo 的 React 完整寫法 
function reducer(state, action) {
  switch (action.type) {
    case "inc":   return { count: state.count + 1 };
    case "dec":   return { count: state.count - 1 };
    case "reset": return { count: 0 };
    default:      return state;
  }
}
function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 });
  return (<>
    <span>{state.count}</span>
    <button onClick={() => dispatch({ type: "inc" })}>+1</button>
    <button onClick={() => dispatch({ type: "dec" })}>−1</button>
    <button onClick={() => dispatch({ type: "reset" })}>reset</button>
  </>);
}

3.6 Context：跨多層傳同一份資料

當很多層、很多元件都要讀同一份「全站級」資料（主題、登入者、語系），一路用 props 傳很痛苦（叫 prop drilling）。Context 讓你在上層 Provider 一次供給，下面任何深度的元件用 useContext 直接拿。

何時用：資料是「很多元件共享的全站設定」。別濫用——不是所有 state 都塞 Context，那會讓每次變動牽動一大片重繪。多數 state 用本地 + 提升就好。

3.7 Reducer + Context = 輕量全域狀態

把 useReducer 的 state/dispatch 放進 Context，就得到一個不用裝任何套件的輕量全域狀態方案：邏輯集中（reducer）、供給廣（context）。多數中小型 App 用這組就夠，不必急著上 Redux/Zustand。

4 · 逃生艙 — 需要跳出 React 模型時

「逃生艙（Escape Hatches）」是你需要碰觸 React 之外的世界（DOM、計時器、網路、第三方 lib）時用的工具。它們很強，但也最容易被誤用——尤其 useEffect。

4.1 ref：存一個「不影響畫面」的值 / 抓 DOM

useRef 給你一個 re-render 之間會保留、但改它不會觸發 render 的盒子。用來存不該影響畫面的東西（計時器 id、上一個值），或抓真實 DOM 節點（focus()、量尺寸、播放影片）。

下面對比 state 和 ref 的差別——注意按 ref 那顆時，畫面「不動」：

互動：ref vs state — 誰會觸發 render？
0次 render state: 0 ref: 0
按 setState+1：畫面立刻更新。按 ref+1：畫面不會動。
</> 展開：這個 demo 的 React 完整寫法 
function RefVsState() {
  const [state, setState] = useState(0);
  const ref = useRef(0);
  console.log("render"); // 只有 setState 會讓元件重跑這行
  return (<>
    <p>state: {state} · ref: {ref.current}</p>
    <button onClick={() => setState(state + 1)}>setState + 1</button>
    <button onClick={() => { ref.current += 1; }}>ref + 1</button>
  </>);
}
const ref = useRef(0);
ref.current += 1;        // 改它 → 不 re-render
inputRef.current.focus(); // 抓 DOM 節點直接操作

4.2 useEffect：跟「外部系統」同步

Effect 是 React 把畫面提交到 DOM 之後才跑的 side effect，專門用來跟 React 以外的系統同步：訂閱事件、開/關計時器、連 WebSocket、操作非 React 的 DOM 函式庫。

useEffect(() => {
  const id = setInterval(tick, 1000); // setup
  return () => clearInterval(id);        // cleanup（一定要收）
}, [/* 依賴：用到的 reactive 值都要列 */]);

依賴陣列要誠實列出 Effect 內用到的每個 props/state，官方 ESLint 規則會幫你抓漏。每次依賴變，React 會先 cleanup 上一輪、再 setup 這一輪——所以清理函式一定要寫（不然計時器、訂閱會越積越多）。

4.3 ★ 你可能不需要 Effect（最重要的一節）

新手最大的坑：把什麼都塞進 useEffect。官方專門寫了一頁「You Might Not Need an Effect」。判斷你要做的事本質屬於哪一種：

下面示範最常見的誤用：把「由 first + last 算出的 fullName」塞進 Effect，會多一次 render（畫面可能閃一下）。打字看 render 次數，勾選框切換兩種寫法：

互動：算得出來的值，不要用 Effect
→ 全名：—
render 次數：0
直接在 render 時算 full = first + last：一次 render 就好。勾起來看 Effect 版怎麼多花一倍。
</> 展開：這個 demo 的 React 完整寫法 
function NameForm() {
  const [first, setFirst] = useState("");
  const [last, setLast] = useState("");
  const full = (first + " " + last).trim(); // ✅ render 時算，不用 Effect
  return (<>
    <input value={first} onChange={e => setFirst(e.target.value)} />
    <input value={last}  onChange={e => setLast(e.target.value)} />
    <b>{full}</b>
  </>);
}
// ❌ 用 Effect 派生 state：多一次 render、容易不同步
const [full, setFull] = useState("");
useEffect(() => { setFull(first + " " + last); }, [first, last]);

// ✅ render 時直接算就好
const full = first + " " + last;

4.4 自訂 Hook：把有狀態的邏輯抽出來重用

當多個元件需要「同一段帶 state/Effect 的邏輯」（例如 useOnlineStatus、useFetch），把它抽成一個 useXxx 函式。自訂 Hook 只是把 built-in Hook 組起來的函式，慣例以 use 開頭，讓 React 認得。

function useOnlineStatus() {
  const [online, setOnline] = useState(navigator.onLine);
  useEffect(() => {
    const on = () => setOnline(true), off = () => setOnline(false);
    window.addEventListener("online", on);
    window.addEventListener("offline", off);
    return () => { /* 記得移除監聽 */ };
  }, []);
  return online;
}
