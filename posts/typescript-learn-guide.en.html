TL;DR

🧠 One-line mental model: TypeScript = JavaScript + a layer of type checking that only exists at compile time.

Types are never shipped or run — tsc checks them and then erases them all, emitting plain JavaScript. So TS helps you while you write (red squiggles, autocomplete, safe refactors), not while it runs. Get that, and everything else clicks.

The recommended order (also this article's chapters): annotations → object shapes (interface/type) → unions & narrowing → function types → generics → the real world → tsconfig & traps. 0 · Before You Start — What TypeScript Is and Why You Want It

0.1 What TypeScript actually is

TypeScript is a superset of JavaScript: any valid JS is valid TS. It does exactly one thing — it adds a type-checking layer while you write. Once it checks out, tsc erases every annotation and emits plain JavaScript that browsers and Node run directly.

★ One crucial takeaway: types do not exist at runtime. So you cannot branch on a type while the code runs ("if x is a User, then…"). To check what a value actually is, use plain JS — typeof / in / Array.isArray (chapter 3).

0.2 Why it's worth it — one example

Same logic. In JS it runs and blows up only when you hit the bad path; in TS the squiggle appears the instant you type:

// JavaScript: perfectly legal, explodes at runtime
function total(cart) {
  return cart.reduce((sum, item) => sum + item.price, 0);
}
total({ price: 100 });     // ❌ at runtime: cart.reduce is not a function (wrong type passed)

// TypeScript: error as you type — you never reach that line
function total(cart: { price: number }[]): number {
  return cart.reduce((sum, item) => sum + item.price, 0);
}
total({ price: 100 });     // ❌ squiggle: Argument of type '{price:number}' is not assignable to '{price:number}[]'
total([{ price: 100 }]);   // ✅ pass an array

What TS really buys you is three things, all before the code runs:

① Catch bugs early
Wrong type, misspelled property, forgotten undefined — squiggled on the spot, not discovered when it crashes.
90% of dumb bugs in big projects are this kind
② Autocomplete
The editor knows every variable's shape, so typing . lists every property/method — no flipping back to docs.
Types are documentation that travels with the code
③ Fearless refactors
Rename a field and every use lights up red; fix until all green and you're done. Without types this is a nightmare.
This is what lets you change big code confidently

1 · Annotations — Telling TS "This Is What This Is"

1.1 How to annotate — but don't, when you don't have to

An annotation is just : Type after a name. But TS is smart — most of the time it figures the type out and you don't annotate at all.

// Annotate variables, params, return values with : Type after the name
let name: string = "Michael";
let age: number = 30;
let ok: boolean = true;

// ✅ but the : string here is redundant — the right side is a string, TS infers it
let city = "Taipei";        // TS infers: city is string
city = 101;                 // ❌ squiggle: Type 'number' is not assignable to 'string'

// Rule: let TS infer on initialization; annotate only when unsure or no initial value
let score;                  // no value → type is any (dangerous, see 1.3)
let score2: number;         // ✅ assigned later, so state it up front

When to annotate vs. let it infer — the thing beginners agonize over. One line settles it:

SituationDoWhy
Has an initial valueDon't — let it inferAnnotating repeats yourself; changing the value means changing two places
Function parametersAlways annotateTS can't guess what you'll feed in
Function return valueRecommendedA wrong return errors here, not somewhere downstream
Declared, assigned laterAnnotateOtherwise it becomes any

1.2 Everyday types: primitives, arrays, objects

// ── Primitives (they line up with JS's typeof) ──
string   number   boolean   null   undefined

// ── Arrays: two spellings, same meaning — pick one and stick with it ──
let nums: number[] = [1, 2, 3];
let tags: Array<string> = ["js", "ts"];   // the generic spelling; chapter 5
nums.push("4");              // ❌ only numbers allowed in here

// ── Objects: just write out the "shape" ──
let user: { name: string; age: number } = { name: "Michael", age: 30 };
user.name;                    // ✅ string, and typing . autocompletes
user.email;                  // ❌ squiggle: no email property on this shape

// ── Tuple: fixed length, fixed type per slot ──
let point: [number, number] = [10, 20];   // coords; React's useState returns one

★ Writing object shapes inline gets long and ugly fast, and you can't reuse it. So real code names shapes with interface / type — the star of chapter 2.

Interactive: inference — what TS figures out from the right side
Pick one to see what TS infers when you don't annotate. const and let infer differently — which is exactly why you often don't annotate by hand.

1.3 Three special types: any, unknown, never

These three are TS-only concepts JS never had. Understanding them is understanding what "safety" means in TS.

// any: turns type checking OFF. TS's escape hatch — and its biggest trap.
let a: any = 42;
a.foo.bar.baz();              // ⚠ TS says nothing; explodes at runtime. any = giving up protection

// unknown: "I don't know yet" — safer than any, must check before use
let u: unknown = fetchData();
u.toUpperCase();             // ❌ squiggle: narrow it first
if (typeof u === "string") {
  u.toUpperCase();            // ✅ confirmed string, safe here
}

// never: "can't happen" — function that never returns, or exhaustiveness checks (ch.3)
function fail(msg: string): never { throw new Error(msg); }

The rule: avoid any whenever you can. When you genuinely don't know a type (say, fresh out of JSON.parse), use unknown and force yourself to check before using it. Treat any as a confession that you gave up right here — the less of it, the better.

2 · Object Shapes — interface and type

2.1 Give a shape a name

Rather than spelling out an object shape every time, name it and reuse it. Two tools: interface and type.

// interface: describes an object shape, the most common tool
interface User {
  name: string;
  age: number;
  email?: string;          // ? = optional, may be absent
  readonly id: number;      // readonly = can't reassign after creation
}

// type: a type alias — can do more than just objects
type User = {
  name: string;
  age: number;
};

// Same usage: use the name as an annotation
function greet(u: User): string {
  return `Hi, ${u.name}`;   // typing u. autocompletes name / age / email / id
}

const me: User = { name: "Michael", age: 30, id: 1 };
me.id = 2;                   // ❌ readonly, can't change

2.2 Structural typing (duck typing) — TS's most important intuition

TS decides whether a type fits by its shape, not its name. If an object has all the properties User needs, it is a User — no need to declare "I am a User". This is structural typing, a.k.a. duck typing.

interface User { name: string; age: number; }

function greet(u: User) { return u.name; }

const a = { name: "A", age: 30 };
greet(a);                     // ✅ shape matches; no "a is User" needed

const b = { name: "B", age: 25, city: "Taipei" };
greet(b);                     // ✅ extra city is fine; everything needed is there

const c = { name: "C" };
greet(c);                    // ❌ age missing, shape too short

★ One exception to know: passing an object literal directly, extra properties ARE rejected (excess property check). greet({ name:"A", age:30, city:"x" }) squiggles — TS is helpfully warning that city might be a typo. Store it in a variable first (like b above) and it passes.

2.3 interface or type? For 90% of cases, either

They overlap heavily — don't agonize. A simple rule: use interface for object shapes, type for everything else.

interfacetype
Describe object shape✓ yes✓ yes
Union A | B✗ no✓ only this one
Name a primitive/tuple✗ no✓ type ID = string
Extendextends& intersection
Auto-merge same nameYes (good for augmenting 3rd-party types)No (duplicate = error)
// Object shapes → interface (can extends)
interface Animal { name: string; }
interface Dog extends Animal { breed: string; }

// Unions, aliases → type only
type Status = "idle" | "loading" | "done";   // the literal union of ch.3
type ID = string | number;
type Dog = Animal & { breed: string };            // & is intersection = merged shape

3 · Unions, Literals & Narrowing — Where TS Shines

3.1 Union types: "this, or that"

Join a few types with | to mean "could be any one of these". It's TS's most expressive feature and the one closest to how real data behaves.

type ID = string | number;      // id could be a string or a number
let id: ID = 101;
id = "A-101";                  // ✅ both allowed
id = true;                   // ❌ boolean isn't in the union

// The most common use: a value that might be absent
function find(id: number): User | undefined {  // returns undefined if not found
  return users.find(u => u.id === id);
}

But a union has one restriction: you can only use what every member is guaranteed to have, because TS doesn't yet know which one it is:

function pad(value: string | number) {
  value.toUpperCase();        // ❌ number has no toUpperCase; not guaranteed
  value.toString();            // ✅ both string and number have toString
}

3.2 Narrowing — the right way to use a union

To use something specific to one member of a union, first prove to TS which one it is. You do that with the checks JS already has — TS understands them and automatically "narrows" the type inside the matching branch.

function pad(value: string | number) {
  if (typeof value === "string") {
    return value.toUpperCase();  // ✅ TS knows value is string here
  }
  return value.toFixed(2);       // ✅ only number can be left
}

3.3 Literal types & discriminated unions — the state-modeling superpower

A type doesn't have to be string — it can be pinned to one specific string value. That's unbeatable for describing "a finite set of states".

// Literal types: can only be one of these strings
type Status = "idle" | "loading" | "success" | "error";
let s: Status = "loading";
s = "loadng";                // ❌ typo caught on the spot — that's the value

// Discriminated union: tell members apart by a shared "tag" field
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "rect"; width: number; height: number };

function area(shape: Shape): number {
  switch (shape.kind) {              // narrow via the kind tag
    case "circle": return Math.PI * shape.radius ** 2;   // ✅ only radius here
    case "rect":   return shape.width * shape.height;      // ✅ only width/height here
  }
}

★ Discriminated unions are TS's best weapon for modeling real data: an API's "success or failure", a Redux / useReducer action, a UI loading state — all fit. See "one value with several mutually exclusive shapes" and reach for it.

4 · Function Types — Annotating Params and Returns

4.1 Parameters and return values

// Always annotate params; annotating the return is recommended (TS confirms you got it right)
function add(a: number, b: number): number {
  return a + b;
}

// Arrow functions the same
const mul = (a: number, b: number): number => a * b;

// Optional ? / default / rest params (just like JS, plus types)
function greet(name: string, greeting: string = "Hi", ...tags: string[]) {
  return `${greeting}, ${name}`;
}
greet("Michael");            // ✅ greeting has a default, tags can be absent

// A function that returns nothing: return type is void
function log(msg: string): void {
  console.log(msg);
}

4.2 Typing "the function itself"

In JS a function is a value — passed as an argument, returned as a result (higher-order functions). In TS that function value has a type too, written (param: Type) => ReturnType.

// Typing a callback — the most common place you annotate a function
function onClick(handler: (event: string) => void) {
  handler("clicked");
}

// Name a function type with type, and reuse it
type Reducer = (acc: number, cur: number) => number;
const sum: Reducer = (a, b) => a + b;   // a, b need no annotation — inferred from Reducer

// Array method callbacks: TS infers the param types for you, usually no annotation
[1, 2, 3].map(n => n * 2);              // n is automatically number, . autocompletes

★ Note above: in const sum: Reducer = (a, b) => …, a and b aren't annotated yet don't error — TS reads them back from the Reducer on the left. This is contextual typing: type info flows from "where it's assigned" into the parameters, which is why so many callbacks need no annotation.

5 · Generics — One Codebase, Safely, for Many Types

Generics feel like the mystical chapter, but the motivation is plain: you want a function that handles many types without giving up type safety (i.e. without any). Generics are "passing the type in as a parameter".

5.1 Why you need generics

// Want a "return whatever you got" function. any? You just threw the type away:
function identity(x: any): any { return x; }
const a = identity(42);       // a is any ❌ no autocomplete, no number checks

// Generics: capture "what type came in" with <T> and carry it to the return unchanged
function identity<T>(x: T): T { return x; }
const b = identity(42);       // b is number ✅ TS infers T = number
const c = identity("hi");     // c is string ✅ same code, each keeps its type
Interactive: generic identity — type in, same type out
identity<T>(x), feed:
Watch TS infer T and carry it to the return type unchanged. Contrast with any: that would throw all the type info away here.

5.2 You use generics daily without noticing

You've been using generics all along — most built-in JS types are generic, the type just gets inferred:

Array<number>                 // same as number[], "an array of numbers"
Promise<User>                 // "a Promise that will give you a User"
Map<string, number>           // a Map with string keys and number values

// In your own generic functions, <T> can be multiple, and can be constrained
function firstKey<T extends object>(obj: T): keyof T {
  return Object.keys(obj)[0] as keyof T;   // T extends object = constrain T to be an object
}

Four built-in utility types you'll use daily (TS's built-in generics that reshape existing types):

Utility typeWhat it doesExample
Partial<T>Make all properties optionalUpdate function args: Partial<User>
Pick<T, K>Keep only a few propertiesPick<User, "name">
Omit<T, K>Drop a few propertiesOmit<User, "id">
Record<K, V>Make a key→value object typeRecord<string, number>

★ You don't need all of generics at once. Early on, reading Array<T> and Promise<T> and using Partial / Pick / Omit is plenty. The need to write your own generic functions usually shows up only when you build shared utilities or libraries.

6 · The Real World — DOM, APIs, Third-Party Packages

6.1 The DOM: TS knows querySelector might find nothing

TS types the DOM APIs for you, so it forces you to face "the element might not exist" — the exact source of countless JS runtime errors.

// querySelector returns Element | null (might find nothing)
const btn = document.querySelector("#save");
btn.addEventListener(...);    // ❌ squiggle: btn could be null

// ✅ Fix 1: check first (safest, preferred)
if (btn) btn.addEventListener("click", ...);

// ✅ Fix 2: type assertion — tell TS "I'm sure it's there, and it's a button"
const input = document.querySelector("#name") as HTMLInputElement;
input.value;                  // ✅ now TS knows about .value (a plain Element has none)

// Non-null assertion ! : "I promise this isn't null" — handy but risky, think first
const el = document.querySelector("#app")!;   // if it really is null, still crashes at runtime

★ as (type assertion) and ! (non-null assertion) are you saying "trust me" to TS. They add no runtime check at all — they just silence the compiler. If the assertion is wrong, it still blows up at runtime. So prefer an if check over an assertion — save assertions for the few cases where you genuinely know more than TS.

6.2 API responses: unknown + guards, don't just trust it

For data off the network, TS can't know its shape at compile time (types don't exist at runtime, remember). fetch's .json() returns any / Promise<any> — a trap that quietly turns type checking off.

async function loadUser(id: number): Promise<User | null> {
  const res = await fetch(`/api/user/${id}`);
  if (!res.ok) return null;

  const data: unknown = await res.json();   // ✅ type it unknown to force validation

  // Confirm the shape with a guard before returning it as a User
  if (isUser(data)) return data;         // ✅ data is narrowed to User after this
  return null;
}

// Type guard: returns `x is User`, so TS narrows based on it
function isUser(x: unknown): x is User {
  return typeof x === "object" && x !== null && "name" in x;
}

★ In practice people use libraries like Zod: write a schema once and get both runtime validation and a TS type, no hand-written guard. But understand the unknown + guard principle first, so the tool isn't magic.

6.3 Third-party types — what @types is

// Some packages ship their own types — install and you have them (most new ones do)
npm install zod              # ships .d.ts; import and you get autocomplete

// Older packages don't; install the community-maintained @types separately
npm install lodash
npm install -D @types/lodash    # type definitions from DefinitelyTyped

If TS complains "could not find a declaration file" when you import a package, nine times out of ten you just need the matching @types/xxx. Only if that still doesn't help do you write your own .d.ts — but that's advanced; ignore it while learning.

7 · tsconfig and Common Traps

7.1 tsconfig: one setting matters most — strict

tsconfig.json has many options, but at first you only need to recognize one: "strict": true. It turns on all the strict checks at once and is the precondition for TS actually protecting you.

{
  "compilerOptions": {
    "strict": true,            // ★ most important: all strict checks on; always on for new projects
    "target": "ES2020",        // which JS version to compile to
    "module": "ESNext",        // which module syntax
    "moduleResolution": "Bundler",
    "noEmit": true            // only check, don't emit (common when a bundler like Vite handles output)
  }
}

★ The most impactful part of strict is strictNullChecks: with it on, null and undefined can no longer slip into any type, forcing you to handle "might be absent" head-on. This one flag stops more bugs than all the rest combined. Never turn strict off to kill a few squiggles.

7.2 Six traps you'll definitely hit

1. "Fixing" types with any
Annoyed by squiggles, you slap on any — that's turning TS off. If you truly don't know the type, use unknown and force a check.
Enable noImplicitAny so any can't sneak in
2. Thinking types exist at runtime
Types are erased after compile — no if (x is User). To check a value, use typeof / in / a guard.
Types are compile-time only; runtime is plain JS
3. Treating as as a cast
as does no runtime check — it just tells TS to hush. A wrong assertion still crashes. Prefer a check when you can.
as means "trust me", not "convert to"
4. Ignoring null / undefined
With strict on, obj.a.b is often blocked because a might be absent. Handle it with ?. and checks, don't force it with !.
The squiggle is helping, not nitpicking
5. Types more complex than the code
Overdo generics and conditional types and nobody can read them. Types exist to clarify, not to show off — keep it simple.
Unreadable types are fresh technical debt
6. Turning strict off for convenience
Disabling strict to dodge a few errors pays TS's cost without taking its benefit. Always on for new projects.
strict off = syntax only, no protection
