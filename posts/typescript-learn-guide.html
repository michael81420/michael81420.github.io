TL;DR

🧠 一句話心智模型：TypeScript = JavaScript + 一層「編譯時才存在」的型別檢查。

型別不會被打包進去跑——tsc 檢查完就把型別全部擦掉，最後產出的是普通 JavaScript。所以 TS 幫你的是寫程式的當下（編輯器紅線、自動補全、重構安全），而不是執行的時候。這點想通，後面全部好懂。

建議的學習順序（也就是本篇的章節）：型別標註 → 物件形狀（interface/type）→ 聯合與窄化 → 函式型別 → 泛型 → 和真實世界打交道 → tsconfig 與坑。先別碰泛型，把前三章的「形狀」概念打穩，第 5 章才不會覺得像天書。

★ 學 TS 最快的方法是在 TypeScript Playground 邊看邊打。它會即時標紅線、告訴你推論出的型別、還能看編譯後的 JS。本篇每段程式碼都可以貼進去看。

0 · 開始之前 — TypeScript 是什麼、為什麼要它

0.1 TypeScript 到底是什麼

TypeScript 是 JavaScript 的超集（superset）：任何合法的 JS 都是合法的 TS。它只做一件事——在你寫程式時，多一層「型別檢查」。檢查完，tsc 會把所有型別標註擦掉，產出的就是瀏覽器／Node 能直接跑的普通 JavaScript。

★ 一個關鍵結論：TS 的型別在執行時完全不存在。所以你不能用型別在跑的時候做判斷（例如「如果 x 是 User 型別就…」），要判斷值長怎樣得靠 JS 本來就有的 typeof／in／Array.isArray（第 3 章）。

0.2 為什麼值得學 — 一個範例就懂

同一段邏輯，JS 要跑起來、真的踩到才知道錯；TS 在你打字的當下就畫紅線：

// JavaScript：完全合法，跑起來才炸
function total(cart) {
  return cart.reduce((sum, item) => sum + item.price, 0);
}
total({ price: 100 });     // ❌ 執行時：cart.reduce is not a function（傳錯型別了）

// TypeScript：打字當下就報錯，根本跑不到那一行
function total(cart: { price: number }[]): number {
  return cart.reduce((sum, item) => sum + item.price, 0);
}
total({ price: 100 });     // ❌ 紅線：Argument of type '{price:number}' is not assignable to '{price:number}[]'
total([{ price: 100 }]);   // ✅ 傳陣列才對

TS 幫你的其實是三件事，且都發生在執行之前：

① 提早抓錯
傳錯型別、拼錯屬性名、忘記處理 undefined——編輯器當場紅線，不用等跑到才發現。
大專案 90% 的低級 bug 都是這類
② 自動補全
編輯器知道每個變數的形狀，打 . 就跳出所有可用屬性／方法，不用回頭翻文件。
型別就是「跟著程式走的文件」
③ 重構有底氣
改一個欄位名字，所有用到的地方立刻標紅，改到全綠就對了。沒有型別時這是惡夢。
敢改大程式，靠的就是這個

0.3 十秒開始寫 — 不用裝任何東西

打開 TypeScript Playground，左邊打 TS、右邊即時看編譯出的 JS，滑鼠移到變數上會顯示型別，錯的地方直接紅線。本篇每段都能貼進去玩。

要在本機的專案裡跑，只有三步：

# ① 裝 TypeScript
npm install -D typescript

# ② 產生設定檔 tsconfig.json（第 7 章會講裡面設什麼）
npx tsc --init

# ③ 編譯：把 .ts 變成 .js
npx tsc            # 或 npx tsc --watch 邊改邊自動編譯

★ 實務上你很少手動 tsc。用 Vite／Next.js 這類工具時，型別檢查是編輯器（VS Code 內建）＋打包工具一起做的。剛學階段，Playground 或 VS Code 開一個 .ts 檔就夠了，別急著配置一堆東西。

1 · 型別標註 — 告訴 TS「這是什麼」

1.1 標註的寫法：冒號加型別，但能不標就不標

型別標註就是在名字後面加 : 型別。但 TS 很聰明——大部分時候它自己推得出來，你不用標。

// 標註：變數、參數、回傳值都在名字後面加 : 型別
let name: string = "Michael";
let age: number = 30;
let ok: boolean = true;

// ✅ 但這行的 : string 是多餘的 —— 右邊是字串，TS 自己就推出來了
let city = "Taipei";        // TS 推論：city 是 string
city = 101;                 // ❌ 紅線：Type 'number' is not assignable to 'string'

// 準則：初始化時讓 TS 推論，不確定或沒初始值時才手動標
let score;                  // 沒給值 → 型別是 any（危險，見 1.3）
let score2: number;         // ✅ 之後才賦值，就先標清楚

什麼時候該標、什麼時候讓它推——這是新手最容易糾結的地方，記住一句話就好：

情況怎麼做為什麼
宣告時就有初始值別標，讓它推標了是重複，改值還要改兩處
函式的參數一定要標TS 推不出你打算餵什麼進來
函式的回傳值建議標寫錯回傳時當場報錯，而不是傳到別處才爆
先宣告後賦值要標否則會變成 any

1.2 常用型別：原始型別、陣列、物件

// ── 原始型別（跟 JS 的 typeof 對得上）──
string   number   boolean   null   undefined

// ── 陣列：兩種寫法，意思一樣，選一種用到底 ──
let nums: number[] = [1, 2, 3];
let tags: Array<string> = ["js", "ts"];   // 泛型寫法，第 5 章會懂
nums.push("4");              // ❌ 陣列裡只能放 number

// ── 物件：直接把「形狀」寫出來 ──
let user: { name: string; age: number } = { name: "Michael", age: 30 };
user.name;                    // ✅ string，打 . 會自動補全
user.email;                  // ❌ 紅線：形狀上沒有 email 這個屬性

// ── 元組 tuple：固定長度、每格型別固定的陣列 ──
let point: [number, number] = [10, 20];   // 座標、React 的 useState 回傳就是這個

★ 物件形狀直接寫在標註裡很快就會變得又臭又長，而且無法重用。所以真實世界都用 interface / type 幫形狀取名字——這是第 2 章的主角。

互動：型別推論 — TS 自己看右邊推出什麼型別
點一個，看 TS 在你沒標註時自己推出什麼型別。const 和 let 推的還不一樣——這正是為什麼常常不用手標。

1.3 三個特殊型別：any、unknown、never

這三個是 TS 特有、JS 沒有的概念，搞懂它們就懂了 TS 的「安全」是什麼意思。

// any：關掉型別檢查。它是 TS 世界的逃生門，也是最大的坑。
let a: any = 42;
a.foo.bar.baz();              // ⚠ TS 完全不擋，跑起來直接炸。用 any = 放棄保護

// unknown：「我還不知道是什麼」——比 any 安全，用之前一定要先確認
let u: unknown = fetchData();
u.toUpperCase();             // ❌ 紅線：得先縮小型別才能用
if (typeof u === "string") {
  u.toUpperCase();            // ✅ 確認過是 string，這裡就安全
}

// never：「不可能發生」——函式永遠不回傳、或窮舉檢查用（第 3 章會再遇到）
function fail(msg: string): never { throw new Error(msg); }

準則：能不用 any 就不用。真的不知道型別（例如剛 JSON.parse 出來的東西）用 unknown，逼自己在使用前先檢查。把 any 當成「我這裡放棄了」的自白，越少越好。

2 · 物件的形狀 — interface 與 type

2.1 幫形狀取個名字

與其每次都把物件形狀寫一長串，不如取個名字重複用。兩種工具：interface 和 type。

// interface：描述物件形狀，最常見
interface User {
  name: string;
  age: number;
  email?: string;          // ? = 可選，可以沒有
  readonly id: number;      // readonly = 只讀，建立後不能改
}

// type：型別別名，能做的更多（不只物件）
type User = {
  name: string;
  age: number;
};

// 用法一樣：拿名字當標註
function greet(u: User): string {
  return `Hi, ${u.name}`;   // 打 u. 會自動補出 name / age / email / id
}

const me: User = { name: "Michael", age: 30, id: 1 };
me.id = 2;                   // ❌ readonly，不能改

2.2 結構型別（鴨子型別）— TS 最重要的直覺

TS 判斷「型別對不對」看的是形狀，不是名字。只要一個物件有 User 需要的所有屬性，它就是 User——不用刻意宣告「我是 User」。這叫結構型別（structural typing），俗稱鴨子型別。

interface User { name: string; age: number; }

function greet(u: User) { return u.name; }

const a = { name: "A", age: 30 };
greet(a);                     // ✅ 形狀對，不用寫「a 是 User」

const b = { name: "B", age: 25, city: "Taipei" };
greet(b);                     // ✅ 多帶 city 也通過，用得到的都在

const c = { name: "C" };
greet(c);                    // ❌ 少了 age，形狀不足

★ 有個例外要知道：直接傳「物件字面量」時，多帶的屬性反而會被擋（叫 excess property check），greet({ name:"A", age:30, city:"x" }) 會紅線。這是 TS 貼心地提醒你「city 可能是打錯的屬性名」。先存進變數再傳（像上面的 b）就不會擋。

2.3 interface 還是 type？九成情況隨便你

兩者高度重疊，別在這糾結太久。一個簡單準則：描述物件用 interface，其他一律用 type。

interfacetype
描述物件形狀✓ 可以✓ 可以
聯合型別 A | B✗ 不行✓ 只有它能
幫原始型別／元組取名✗ 不行✓ type ID = string
延伸extends& 交集
同名自動合併會（適合擴充第三方型別）不會（重複定義報錯）
// 描述物件 → interface（可 extends 繼承）
interface Animal { name: string; }
interface Dog extends Animal { breed: string; }

// 聯合、別名 → 只能用 type
type Status = "idle" | "loading" | "done";   // 第 3 章的字面量聯合
type ID = string | number;
type Dog = Animal & { breed: string };            // & 是交集，等於合併形狀

3 · 聯合、字面量與窄化 — TS 最好用的地方

3.1 聯合型別：「是這個，或那個」

用 | 把幾個型別串起來，表示「可能是其中任一個」。這是 TS 表達力最強、也最貼近真實資料的功能。

type ID = string | number;      // id 可能是字串或數字
let id: ID = 101;
id = "A-101";                  // ✅ 兩種都行
id = true;                   // ❌ boolean 不在聯合裡

// 最常見用途：一個值可能沒有
function find(id: number): User | undefined {  // 找不到就回 undefined
  return users.find(u => u.id === id);
}

但聯合型別有個限制：只能用「每個成員都保證有」的東西。因為 TS 不知道現在到底是哪一個：

function pad(value: string | number) {
  value.toUpperCase();        // ❌ number 沒有 toUpperCase，不能保證
  value.toString();            // ✅ string 和 number 都有 toString
}

3.2 型別窄化 — 聯合型別的正確用法

要用聯合型別裡「某一種特有」的東西，得先讓 TS 確定現在是哪一種。方法就是用 JS 本來就有的判斷——TS 看得懂這些判斷，會在對應的分支裡自動把型別「窄化」成更具體的那個。

function pad(value: string | number) {
  if (typeof value === "string") {
    return value.toUpperCase();  // ✅ 這裡 TS 確定 value 是 string
  }
  return value.toFixed(2);       // ✅ 剩下的只可能是 number
}

四種最常用的窄化工具，全都是 JS 原生語法，TS 只是「看得懂」：

工具用來窄化例子
typeof原始型別typeof x === "string"
in物件有沒有某屬性"swim" in animal
instanceof是不是某 class 的實例err instanceof Error
=== 字面量判別聯合（見 3.3）shape.kind === "circle"
互動：窄化 — 同一個聯合值，走進哪個分支
value: string | number | boolean，餵：
點一個值，看 typeof 判斷後 TS 把它窄化成哪個具體型別，以及在那個分支裡哪些方法變成安全可用。

3.3 字面量型別與判別聯合 — 狀態管理的神器

型別不只可以是 string，還可以精確到某個特定的字串值。這在描述「有限的幾種狀態」時無敵好用。

// 字面量型別：只能是這幾個字串之一
type Status = "idle" | "loading" | "success" | "error";
let s: Status = "loading";
s = "loadng";                // ❌ 拼錯字當場被抓，這就是它的價值

// 判別聯合（discriminated union）：用一個共同的「標籤」欄位區分
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "rect"; width: number; height: number };

function area(shape: Shape): number {
  switch (shape.kind) {              // 用 kind 這個標籤窄化
    case "circle": return Math.PI * shape.radius ** 2;   // ✅ 這裡只有 radius
    case "rect":   return shape.width * shape.height;      // ✅ 這裡只有 width/height
  }
}

★ 判別聯合是 TS 建模真實資料的最佳武器：API 回傳的「成功 or 失敗」、Redux／useReducer 的 action、UI 的載入狀態，全都適合。看到「一個值有好幾種互斥的形態」，就想到它。

4 · 函式型別 — 標好參數與回傳

4.1 參數與回傳值

// 參數一定要標，回傳值建議標（讓 TS 幫你確認寫對）
function add(a: number, b: number): number {
  return a + b;
}

// 箭頭函式一樣
const mul = (a: number, b: number): number => a * b;

// 可選參數 ? / 預設值 / 其餘參數（跟 JS 一樣，只是多了型別）
function greet(name: string, greeting: string = "Hi", ...tags: string[]) {
  return `${greeting}, ${name}`;
}
greet("Michael");            // ✅ greeting 有預設、tags 可以沒有

// 不回傳東西的函式：回傳型別是 void
function log(msg: string): void {
  console.log(msg);
}

4.2 把「函式本身」的型別寫出來

函式在 JS 裡是值，可以當參數傳、被回傳（高階函式）。在 TS 裡，這個「函式值」也有型別，寫法是 (參數: 型別) => 回傳型別。

// 幫 callback 標型別：這是標註函式最常見的場合
function onClick(handler: (event: string) => void) {
  handler("clicked");
}

// 用 type 幫函式型別取名，重複用
type Reducer = (acc: number, cur: number) => number;
const sum: Reducer = (a, b) => a + b;   // a、b 不用再標，型別從 Reducer 推來

// 陣列方法的 callback：TS 自動幫你推好參數型別，通常不用標
[1, 2, 3].map(n => n * 2);              // n 自動是 number，打 n. 有補全

★ 注意上面 const sum: Reducer = (a, b) => …，a 和 b 沒有標型別卻不報錯——因為 TS 從左邊的 Reducer 反推出來了。這叫上下文推論：型別資訊會從「被賦值的地方」流進函式參數，所以很多 callback 你根本不用標。

5 · 泛型 — 一份程式碼，安全地吃多種型別

泛型（generics）是新手覺得最玄的一關，但動機其實很單純：你想寫一個能處理多種型別的函式，又不想放棄型別安全（用 any）。泛型就是「把型別當成參數傳進去」。

5.1 為什麼需要泛型

// 想寫一個「回傳傳進來的東西」的函式。用 any？型別就丟了：
function identity(x: any): any { return x; }
const a = identity(42);       // a 是 any ❌ 打 a. 沒補全，數字方法也不檢查

// 泛型：用 <T> 把「進來什麼型別」記住，原封不動帶到回傳
function identity<T>(x: T): T { return x; }
const b = identity(42);       // b 是 number ✅ TS 自動推 T = number
const c = identity("hi");     // c 是 string ✅ 同一份程式，型別各自保留
互動：泛型 identity — 進什麼型別，出什麼型別
identity<T>(x)，餵：
看 TS 怎麼自動推出 T，並把它原封不動帶到回傳型別。對比 any：any 這裡會把型別資訊全丟掉。

5.2 你天天在用泛型，只是沒發現

其實你早就在用泛型了——JS 內建型別大多是泛型的，只是型別自動被推出來：

Array<number>                 // 就是 number[]，「裝 number 的陣列」
Promise<User>                 // 「未來會給你一個 User 的 Promise」
Map<string, number>           // key 是 string、value 是 number 的 Map

// 自己寫泛型函式時，<T> 也能有多個、也能加限制
function firstKey<T extends object>(obj: T): keyof T {
  return Object.keys(obj)[0] as keyof T;   // T extends object = 限制 T 一定是物件
}

四個天天會用到的內建工具型別（TS 內建的泛型，拿現有型別加工出新型別）：

工具型別做什麼例子
Partial<T>把所有屬性變可選更新函式的參數：Partial<User>
Pick<T, K>只挑幾個屬性Pick<User, "name">
Omit<T, K>排除幾個屬性Omit<User, "id">
Record<K, V>做一個 key→value 的物件型別Record<string, number>

★ 泛型不用一次全懂。剛學階段，看得懂 Array<T>、Promise<T>、會用 Partial／Pick／Omit 就夠了。自己寫泛型函式的需求，通常要到你在做共用工具或函式庫時才出現。

6 · 和真實世界打交道 — DOM、API、第三方套件

6.1 DOM：TS 知道 querySelector 可能找不到

TS 幫 DOM API 都標好了型別，所以它會逼你面對「元素可能不存在」——這正是很多 JS 執行時錯誤的來源。

// querySelector 的回傳型別是 Element | null（可能找不到）
const btn = document.querySelector("#save");
btn.addEventListener(...);    // ❌ 紅線：btn 可能是 null

// ✅ 解法一：先檢查（最安全，首選）
if (btn) btn.addEventListener("click", ...);

// ✅ 解法二：型別斷言，告訴 TS「我確定它在、而且是 button」
const input = document.querySelector("#name") as HTMLInputElement;
input.value;                  // ✅ 現在 TS 知道有 .value（一般 Element 沒有）

// 非空斷言 ! ：「我保證這裡不是 null」——方便但有風險，用前想清楚
const el = document.querySelector("#app")!;   // 若真的是 null，執行時還是會炸

★ as（型別斷言）和 !（非空斷言）是你在對 TS 說「相信我」。它們不會產生任何執行時檢查，只是關掉編譯器的抱怨。斷言錯了，執行時一樣爆。所以能用 if 檢查就別用斷言——把斷言留給「你比 TS 更確定」的少數場合。

6.2 處理 API 回傳：unknown + 守衛，別直接信

從網路來的資料，TS 無法在編譯時知道它長怎樣（型別在執行時不存在，記得嗎）。fetch 的 .json() 回傳的是 any／Promise<any>，這是個陷阱——等於悄悄關掉了型別檢查。

async function loadUser(id: number): Promise<User | null> {
  const res = await fetch(`/api/user/${id}`);
  if (!res.ok) return null;

  const data: unknown = await res.json();   // ✅ 標成 unknown，逼自己驗證

  // 用守衛確認形狀對，才敢當成 User 回傳
  if (isUser(data)) return data;         // ✅ 這之後 data 被窄化成 User
  return null;
}

// 型別守衛：回傳 `x is User`，TS 就會據此窄化型別
function isUser(x: unknown): x is User {
  return typeof x === "object" && x !== null && "name" in x;
}

★ 實務上大家會用 Zod 這類函式庫，寫一次 schema 就同時得到「執行時驗證」和「TS 型別」，不用手刻守衛。但先懂 unknown + 守衛的原理，再用工具才不會把它當魔法。

6.3 第三方套件的型別 — @types 是什麼

// 有些套件自帶型別，裝了就有（現在多數新套件都這樣）
npm install zod              # 自帶 .d.ts，import 進來就有補全

// 老套件沒自帶型別，要另外裝社群維護的 @types
npm install lodash
npm install -D @types/lodash    # DefinitelyTyped 提供的型別定義

如果 import 一個套件時 TS 抱怨「找不到型別宣告」，九成就是要裝對應的 @types/xxx。裝了還是沒有，才需要自己寫 .d.ts——但那是進階需求，剛學不用管。

7 · tsconfig 與常見的坑

7.1 tsconfig：一個設定值最重要 — strict

tsconfig.json 選項很多，但剛開始你只需要認得一個："strict": true。它一次打開所有嚴格檢查，是 TS 能真正保護你的前提。

{
  "compilerOptions": {
    "strict": true,            // ★ 最重要：打開全部嚴格檢查，新專案一律開
    "target": "ES2020",        // 編譯成哪個 JS 版本
    "module": "ESNext",        // 用哪種模組語法
    "moduleResolution": "Bundler",
    "noEmit": true            // 只檢查不產出（交給 Vite 等工具打包時常這樣）
  }
}

★ strict 裡面最有感的是 strictNullChecks：打開後 null 和 undefined 不再能隨便塞進任何型別，逼你正面處理「可能沒有值」的情況。這一條擋掉的 bug 比其他全部加起來還多。千萬別為了少幾條紅線把 strict 關掉。

7.2 六個一定會踩到的坑

1. 用 any 把型別「修掉」
紅線煩就標 any——等於把 TS 關掉。真的不知道型別用 unknown，逼自己檢查。
開 noImplicitAny，不讓 any 偷偷出現
2. 以為型別執行時還在
型別編譯後就擦掉了，不能 if (x is User)。要判斷值得用 typeof／in／守衛。
型別只在編譯時，執行時是普通 JS
3. 斷言 as 當成轉型
as 不做任何執行時檢查，只是叫 TS 閉嘴。斷言錯了照樣炸。能檢查就別斷言。
as 是「相信我」，不是「轉成」
4. 忽略 null / undefined
開了 strict 後 obj.a.b 常被擋，因為 a 可能沒有。用 ?. 和先檢查來處理，別用 ! 硬壓。
紅線是在幫你，不是找碴
5. 型別寫得比程式還複雜
泛型、條件型別玩過頭，沒人看得懂。型別是為了好懂，不是炫技，簡單能表達就好。
看不懂的型別 = 新的技術債
6. 關掉 strict 圖方便
為了少報幾個錯把 strict 關掉，等於花了 TS 的成本卻不拿它的好處。新專案一律開。
strict off = 只有語法沒有保護

8.2 接下來學什麼

方向學什麼什麼時候
寫前端React + TypeScript（props、state 都標型別）會了基礎，馬上就想用在真實專案

★ 學 TS 的最好方式是直接用在專案裡，不要空學語法。強烈建議搭配本站的 React 完全上手——React + TS 是現在前端最主流的組合，兩篇一起看，型別和框架的坑會同時通。JS 底子還不穩的話，先補 JavaScript 從零到會用，TS 的型別是加在 JS 之上的，JS 沒懂型別也標不對。
